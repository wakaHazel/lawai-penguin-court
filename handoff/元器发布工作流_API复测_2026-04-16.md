# 元器发布工作流 API 复测

时间：2026-04-16

测试方式：

- 不走公开发布页
- 直接调用元器 OpenAPI：
  - `https://open.hunyuan.tencent.com/openapi/v1/agent/chat/completions`
- 使用已提供的发布凭据：
  - `assistant_id = 2043273425069278272`
  - 已提供 `api_key`

## 结论

这次已经确认：

1. 发布 assistant 本身可访问，OpenAPI 链路是通的。
2. 但主工作流 `W00_企鹅法庭主控编排` 在真正被触发后，仍然会报：

```json
{
  "error": {
    "message": "工作流运行异常",
    "type": "runtime_error",
    "code": "10003"
  }
}
```

所以当前不是“发布页问题”，而是**发布工作流本体运行异常**。

---

## 复测结果

### 1. 最小闲聊请求

请求：

- `messages = 请只回复ok`
- `custom_variables = {}`

结果：

- HTTP `200`
- 返回普通 assistant 文本

说明：

- `api_key`
- `assistant_id`
- OpenAPI endpoint
- 发布 assistant 基础访问

这四项都正常。

### 2. 中文工作流触发词，无变量

触发词：

`执行企鹅法庭当前回合主控编排，并直接返回 result_json。`

变量：

- `{}` 或仅 `case_id`

结果：

- HTTP `200`
- assistant 返回提示，大意是：
  - 当前阶段参数不在预设范围内
  - 请检查 `current_stage` 是否为 `prepare / investigation / evidence / debate / report_ready`

说明：

- 这一步已经不是普通闲聊了，说明工作流触发逻辑在响应。
- 也说明 assistant/工作流能识别 `current_stage` 是关键变量。

### 3. 只要带上 `current_stage`

测试：

- `case_id + current_stage=prepare`
- `case_id + current_stage=investigation`
- `case_id + current_stage=evidence`
- `case_id + current_stage=debate`
- `case_id + current_stage=report_ready`

结果：

- 全部 HTTP `500`
- 全部报：

```json
{
  "error": {
    "message": "工作流运行异常",
    "type": "runtime_error",
    "code": "10003"
  }
}
```

说明：

- 不是某个阶段值写错了；
- 也不是某个后续变量单独触发；
- 而是**工作流一旦真正根据 `current_stage` 进入分支执行，就在内部炸掉**。

### 4. 全量 W00 变量

变量集：

- `case_id`
- `current_stage`
- `selected_action`
- `round_number`
- `v_case_summary`
- `v_case_type`
- `v_focus_issues`
- `v_fact_keywords`
- `v_opponent_role`
- `v_opponent_arguments`
- `v_historical_dialogs`

结果：

- HTTP `500`
- 同样是 `runtime_error / 10003 / 工作流运行异常`

说明：

- 当前后端发送的变量集合并没有解决问题；
- 根因仍在元器工作流内部，而不是本地 API 封装层。

---

## 当前判断

发布态现状可以准确表述为：

```text
发布 assistant 可调用；
工作流能被触发；
但 W00 一旦进入 current_stage 分支执行，就发生工作流内部运行异常。
```

也就是说：

- 不是前端问题
- 不是公开页面问题
- 不是 API key / assistant_id 配置问题
- 不是网络问题
- 而是元器工作流内部逻辑问题

---

## 最值得优先检查的元器工作流位置

让做工作流的同学优先检查：

1. `W00_企鹅法庭主控编排` 的开始节点之后，凡是直接读取 `current_stage` 的条件分支。
2. 所有按 `prepare / investigation / evidence / debate / report_ready` 分流的条件节点。
3. 分流后第一个大模型节点、代码节点、变量映射节点。
4. 是否有节点引用了未定义变量。
5. 是否有某个分支要求额外变量，但开始节点没有兜底默认值。
6. 是否把字符串变量当 JSON、对象或数组直接读取。

---

## 对本地后端的影响

当前本地后端如果继续调用这版发布工作流，会进入降级：

- 捕获 `HTTP 500`
- 记为 `yuanqi_call_failed`
- 返回本地 mock / 退化结果

所以如果你看到平台里只是退化运行，不代表本地代码错了，而是发布工作流本身没跑通。
