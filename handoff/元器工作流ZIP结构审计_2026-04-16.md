# 元器工作流 ZIP 结构审计

审计时间：2026-04-16  
审计对象：`tmp/yuanqi_export_W00`  
审计脚本：`scripts/yuanqi_workflow_zip_audit.py`

## 结论

这份 zip 里不是只有一个问题，而是至少有三类结构性风险叠在一起：

1. `W00` 和 `W01` 都把 `round_number` 声明成了 `INT`。
2. `W00` 的 `大模型2`、`大模型3` 以及 `W04` 的 `大模型1` 没有声明输出 schema，但后续节点又在引用它们的 `Output.Content`。
3. `W02` 依赖外部得理插件接口，而且插件请求头里写死了 `appid/secret`。

按当前 zip 来看：

- `prepare` 分支的高概率根因是 `round_number` 的类型契约问题。
- `investigation / evidence / debate` 分支除了 `round_number` 风险，还一定会撞上 `W00.大模型2` 无输出 schema 的问题。
- `report_ready` 分支一定会撞上 `W04.大模型1` 和 `W00.大模型3` 的输出 schema 问题，同时还依赖 `W02` 插件可用。

## 逐项发现

### 1. `round_number` 类型风险

涉及文件：

- `tmp/yuanqi_export_W00/e1256571-6359-4ce1-a4a1-d22c2981ead3_workflow.json`
- `tmp/yuanqi_export_W00/c960a070-d34a-41c0-a7ca-7fb5585e27a6_workflow.json`

具体表现：

- `W00_企鹅法庭主控编排` 的开始节点把 `round_number` 定义为 `INT`
- `W01_庭审场景生成` 的开始节点也把 `round_number` 定义为 `INT`
- `W00` 在调用 `W01` 时继续把 `round_number` 作为 `INT` 透传

为什么危险：

- OpenAPI `custom_variables` 实际上传入时是字符串形态
- 之前 live 复测里，直接把 API 层变量改成原生整数会触发请求参数错误
- 所以当前契约是“API 只能稳传字符串，但工作流硬要求 INT”

这会直接影响：

- `prepare`
- `investigation`
- `evidence`
- `debate`

### 2. `W00` 中存在“引用不存在的 Output.Content”

涉及文件：

- `tmp/yuanqi_export_W00/e1256571-6359-4ce1-a4a1-d22c2981ead3_workflow.json`

具体表现：

- `大模型2` 的 `Outputs` 是空数组
- `大模型3` 的 `Outputs` 是空数组
- 但后续节点还在引用它们的 `Output.Content`

明确的坏引用有四处：

- `回复11.B_output -> 大模型2:Output.Content`
- `回复12.C_output -> 大模型3:Output.Content`
- `代码1.b_text -> 大模型2:Output.Content`
- `代码1.c_text -> 大模型3:Output.Content`

这意味着：

- `investigation / evidence / debate` 路径一旦跑到 `大模型2` 之后，结构上就已经不闭合
- `report_ready` 路径一旦跑到 `大模型3` 之后，也存在同类问题

### 3. `W04` 也有同类输出 schema 缺失

涉及文件：

- `tmp/yuanqi_export_W00/e8259bb4-a3d4-481b-84b6-d9c7cc2ff70d_workflow.json`

具体表现：

- `W04_胜诉率分析与复盘` 的 `大模型1` 的 `Outputs` 是空数组
- 但 `回复1.report` 仍在引用 `大模型1:Output.Content`

明确坏引用：

- `回复1.report -> 大模型1:Output.Content`

这意味着：

- `report_ready` 分支即便前面的 `W02 / W03` 都成功，到了 `W04` 自己也会因为输出引用不闭合而有很高概率报运行时错误

### 4. `W02` 是强外部依赖

涉及文件：

- `tmp/yuanqi_export_W00/068acc40-bce9-4ff9-8b63-8df0673eec96_workflow.json`

插件节点：

- `search_law1 -> POST https://openapi.delilegal.com/api/qa/v3/search/queryListLaw`
- `search_case1 -> POST https://openapi.delilegal.com/api/qa/v3/search/queryListCase`

风险点：

- 请求头写死了 `appid`
- 请求头写死了 `secret`
- 只要得理侧鉴权、额度、白名单、插件发布状态有一个不对，`W02` 就会失败

这会直接影响：

- `report_ready`
- `investigation / evidence / debate` 路径里的 `W02_法律支持检索2`

## 分支级判断

### `prepare`

当前最可疑根因：

- `round_number: INT` 契约问题

原因：

- 这条分支不经过 `W02 / W03 / W04`
- 所以它最像是被 `W01` 的必填 `INT round_number` 卡死

### `investigation / evidence / debate`

当前至少有两层风险：

1. `W01.round_number` 类型风险
2. `W00.大模型2` 无输出 schema

所以这条路径不是“可能有问题”，而是结构上就已经不稳。

### `report_ready`

当前至少有三层风险：

1. `W02` 插件依赖
2. `W04.大模型1` 无输出 schema
3. `W00.大模型3` 无输出 schema

这条路径是当前 zip 里最容易炸的一条。

## 建议修复顺序

1. 先把 `W00`、`W01` 里的 `round_number` 全部统一成 `STRING`
2. 给 `W00.大模型2`、`W00.大模型3`、`W04.大模型1` 重新补齐标准输出 schema
3. 检查所有引用 `Output.Content` 的节点，确认被引用节点真的声明了 `Output -> Content`
4. 再单独验证 `W02` 的两个插件接口是否还能在当前环境正常返回
