## 1. 一句话结论

当前这版工作流不是“参数还没传对”这么简单，而是“进入真实分支后，工作流内部执行链本身仍然有结构性问题”。

1. `W00` 里用了一个公共汇总代码节点 `代码1`，它同时引用了三个互斥分支的大模型输出。

## 3. 当前最可疑的主因

## 3.1 主因一：`W00.代码1` 是一个高风险的公共汇总节点

对应文件：

- [6184ce77-a196-4111-a595-1ffba3383b0c_workflow.json](E:\lawai\tmp\yuanqi_export_W04_2026-04-17\6184ce77-a196-4111-a595-1ffba3383b0c_workflow.json)

当前 `W00` 的三条真实分支是：

1. `prepare`
   - `W01_庭审场景生成1 -> 大模型1 -> 回复1 -> 代码1`
2. `investigation / evidence / debate`
   - `W01_庭审场景生成11 -> W02_法律支持检索2 -> W03_对方行为模拟2 -> 大模型2 -> 回复11 -> 代码1`
3. `report_ready`
   - `W02_法律支持检索1 -> W03_对方行为模拟1 -> W04_胜诉率分析与复盘1 -> 大模型3 -> 回复12 -> 代码1`

但是公共代码节点 `代码1` 的输入是：

1. `a_text <- 大模型1.Output.Content`
2. `b_text <- 大模型2.Output.Content`
3. `c_text <- 大模型3.Output.Content`

问题就在这里：

1. 这三条分支是互斥的。
2. 一次运行只会命中其中一条。
3. 但 `代码1` 仍然同时引用三条分支的大模型输出。

这会导致一个典型风险：

1. 当前分支没有执行到的 LLM 输出，在发布态里可能被当成不存在。
2. 公共节点如果仍去解析这些未执行分支的引用，就可能直接触发运行时错误。

这就是目前最像“所有真实分支一进来就炸”的根因。

### 更直白一点

不是说 `代码1` 一定逻辑写错了，而是它的接法本身就很危险。  

如果平台对“未执行分支的引用”处理不够宽容，这种公共汇总写法就很容易在发布态炸掉。

### 为什么这是危险信号

编辑器里“看起来节点有输出”和导出 JSON 里“正式声明了输出 schema”不是一回事。  

发布态更有可能依赖正式 schema。  
如果 schema 是空的，但后面又引用 `Output.Content`，就很容易出现：

1. 编辑态能看见结果
2. 发布态一跑就炸

这和你现在遇到的现象高度吻合。

## 3.3 主因三：`W02` 的得理插件链路仍然是外部依赖风险点

`W02` 里仍然有两个外部调用节点：

1. `search_law1`

2. `search_case1`
   - `POST https://openapi.delilegal.com/api/qa/v3/search/queryListCase`

这类节点即使编辑态能调通，发布态也仍然可能因为以下任一原因出问题：

1. 插件发布态鉴权和编辑态不一致
2. Header 配置没随发布生效
3. 调用额度问题
4. 平台白名单问题
5. 插件节点的返回结构和下游预期不一致

但从目前现象看，它更像“次一级排查点”，不是第一主因。  
因为 `prepare` 分支理论上不需要经过 `W02`，结果也同样炸了。

这进一步说明：`W02` 不是唯一根因。

## 4. 现在不应该再误判成什么

下面这些点，当前可以明确说“不是第一主因”：

1. 不是 `round_number` 类型没改
   - 最新包里已经是 `STRING`
2. 不是 `current_stage` 根本没传进去
   - fallback 能工作，说明分支识别是生效的
3. 不是“只要把所有参数再多传一点就好了”
   - 全量变量一起传也还是 `500 / 10003`
4. 不是变量完全没映射
   - 关键映射已经存在

## 5. 修复优先级

建议按这个顺序修，不然很容易返工：

1. 先修 `W00` 的公共汇总结构
2. 再补齐 `W00.大模型2 / 大模型3 / W04.大模型1` 的输出 schema
3. 再单测 `W01 / W04 / W02 / W03`
4. 最后再做 `W00` 全链路联调

不要先一遍遍全链路试。  
现在这样会反复看到 `500 / 10003`，但拿不到真正问题点。

## 6. 最推荐的修法

## 6.1 修法 A：取消 `W00.代码1` 的三路公共汇总

这是我最推荐的修法。  

不要再让一个公共节点同时读取 `大模型1 / 大模型2 / 大模型3` 的输出。

改成三条分支各自收口：

1. `prepare` 分支
   - `大模型1 -> 分支结束节点 / 分支代码节点 -> W00.end`
2. `investigation/evidence/debate` 分支
   - `大模型2 -> 分支结束节点 / 分支代码节点 -> W00.end`
3. `report_ready` 分支
   - `大模型3 -> 分支结束节点 / 分支代码节点 -> W00.end`

### 最简单的做法

每个分支各自做一个代码节点：

1. `代码_prepare`
   - 只接 `大模型1.Output.Content`
2. `代码_mid`
   - 只接 `大模型2.Output.Content`
3. `代码_report`
   - 只接 `大模型3.Output.Content`

每个代码节点都输出统一结构：

1. `final_output`
2. `branch_name`

然后分别接到 `W00.end`。

这样可以彻底避免“公共节点引用未执行分支输出”的问题。

## 6.2 修法 B：如果你们坚持保留公共节点

那就必须满足两个前提：

1. `代码1` 内部要对 `a_text / b_text / c_text` 的缺失做非常严格的空值兜底
2. 平台本身必须允许未执行分支的引用传空值而不是直接抛错

这个方案不稳。  
因为现在已经出现发布态统一炸分支的现象，说明平台大概率不够宽容。

所以仍然不推荐。

## 7. 保姆级排查方法

下面这套流程是给元器工作流同学直接照着做的。  
目标不是“边猜边改”，而是一步一步缩小范围。

## 7.1 第一步：不要先调主控 `W00`

先停掉“直接调主控工作流试全链路”的做法。  

先分别单测：

1. `W01_庭审场景生成`
2. `W04_胜诉率分析与复盘`
3. `W02_法律支持检索`
4. `W03_对方行为模拟`

原因：

1. `W00` 太长，信息被吞了
2. 现在统一报 `10003`，不利于定位
3. 先把子工作流一层层跑通，再接回主控，最快

## 7.2 第二步：先查每个大模型节点有没有正式输出

在元器里操作：

1. 打开子工作流或主工作流
2. 点击目标大模型节点
3. 看右侧配置面板
4. 找“输出”“出参”“返回值”“Schema”这类区域
5. 确认不是空的

这一步要检查三个节点：

1. `W00.大模型2`
2. `W00.大模型3`
3. `W04.大模型1`

必须补成至少下面这套：

```text
Output
  Thought: STRING
  Content: STRING
```

如果界面里已经能看到文本输出，但 schema 区域仍然是空的，还是要补。  
不要以“编辑器里能看到返回结果”代替正式 schema。

## 7.3 第三步：单测 `W04`

先测 `W04`，因为它最短，最容易看清问题。

`W04` 的输入只要给四个字符串即可：

1. `v_case_profile`
2. `v_legal_support_summary`
3. `v_simulation_timeline`
4. `v_opponent_behavior`

### 在元器里怎么测

1. 打开 `W04_胜诉率分析与复盘`
2. 点“调试”或“运行”
3. 给四个输入都填上简单纯文本，不要先塞大段复杂 JSON
4. 先确认 `大模型1` 节点能出结果
5. 再确认 `结束` 节点能正确拿到 `Output.report_payload`

### 验收标准

1. `大模型1` 运行成功
2. `结束` 不报错
3. `Output.report_payload` 有实际内容

如果 `W04` 单独都炸，那先不要碰 `W00`。

## 7.4 第四步：单测 `W01`

给 `W01` 填最小可用输入：

1. `current_stage = prepare`
2. `v_case_summary = 任意简短案情`
3. `round_number = 1`
4. `v_historical_dialogs = 空字符串`

验收标准：

1. `scene_payload` 能稳定输出
2. 不要求一开始内容多精美，先要求稳定

## 7.5 第五步：单测 `W02`

这一步只看插件链路，不看主控。

输入建议：

1. `case_id = CASE-001`
2. `case_type = 劳动争议`
3. `focus_issues_json = ["劳动关系是否成立"]`
4. `fact_keywords_json = ["工资流水","考勤记录"]`

### 重点看什么

1. `search_law1` 是否成功
2. `search_case1` 是否成功
3. 它们的返回 body 是否真的有数据
4. 最终 `legal_payload` 是否生成

### 如果这里炸了

优先排查：

1. 插件 header 里的 `appid/secret`
2. 插件发布状态
3. 插件接口是否在发布态可用
4. 返回结构是否变了

## 7.6 第六步：单测 `W03`

给它最小可用输入：

1. `v_opponent_role = defendant`
2. `v_current_stage = investigation`
3. `v_selected_action = 对证据真实性提出异议`
4. `v_likely_arguments = ["双方并非劳动关系"]`
5. `v_focus_issues = ["劳动关系是否成立"]`

验收标准：

1. `opponent_payload` 有输出
2. 输出结构稳定

## 7.7 第七步：回到 `W00`，先删公共汇总，再重接

这一步是关键。

### 具体操作

1. 打开 `W00_企鹅法庭主控编排`
2. 找到 `代码1`
3. 先不要直接改里面代码
4. 先看它是不是同时接了：
   - `大模型1.Output.Content`
   - `大模型2.Output.Content`
   - `大模型3.Output.Content`
5. 如果是，就按下面方式改

### 推荐改法

1. 删掉公共 `代码1`
2. 新建三个分支专属代码节点
   - `代码_prepare`
   - `代码_mid`
   - `代码_report`
3. 每个代码节点只接当前分支自己的大模型输出
4. 每个代码节点都输出：
   - `final_output`
   - `branch_name`
5. 三个代码节点分别连到 `结束`

### 每个代码节点的逻辑可以非常简单

```javascript

  final_output: input_text || "",
  branch_name: "prepare"

```

中间分支和报告分支同理，只改 `branch_name`。

## 8. 如果修完还是炸，下一步查什么

如果你们已经做了下面两件事：

1. 去掉了 `W00` 的公共三路汇总
2. 补齐了缺失 LLM schema

但发布态还炸，那下一优先级是：

1. `W02` 插件调用在发布态是否真的可用
2. `W03` 输出内容是否超过平台限制
3. `W04` 的 Markdown 输出是否过长
4. 某个节点在编辑态能容忍空输入，但发布态不容忍

这时建议改成“二分法”：

1. 先让 `prepare` 单独跑通
2. 再让 `report_ready` 单独跑通
3. 最后再接中间分支
