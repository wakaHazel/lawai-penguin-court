# 元器受阻期间并行推进总表

> 更新日期：2026-04-08  
> 项目：`企鹅法庭·沉浸式庭审模拟与立法沙盘推演AI智能体系统`

## 1. 先说结论

现在真正卡住的，不是“业务逻辑没想清楚”，而是 `元器导出包的变量面板编码方式还没被真实样例锁定`。

所以从现在开始，最稳的推进策略不是继续盲修 zip，而是：

1. 一边拿 `元器最小真实导出样例`
2. 一边把所有不依赖导入成功的工作并行推进

这样后面即使工作流恢复，也不会大面积返工。

## 2. 当前硬阻塞

### 阻塞点 A：元器变量面板无法靠猜测稳定恢复

- 症状：导入后工作流为空、变量面板不显示、启动输入变量缺失、回复节点引用变量失效。
- 当前判断：缺一个真实的、在元器 UI 里手工配置并成功导出的最小样例，来反推导出格式。
- 结论：暂停继续盲猜 `变量.xlsx / 参数.xlsx / workflow.json` 的编码。

### 阻塞点 B：得理工具不会随着 zip 自动创建

- 症状：即使导入工作流，工具节点对应的插件工具仍然需要在元器里手工配置。
- 结论：工作流导入和工具配置是两条线，必须分开做。

## 3. 现在可以并行推进的四条主线

## 3.1 主线一：元器最小真实样例回传

### 目标

拿到一个最小、真实、可导出的元器工作流样例，用来锁定变量面板的真实导出格式。

### 谁来做

- 技术同学
- 或负责元器操作的朋友

### 需要做什么

在元器 UI 里手工新建一个最小工作流，并确保以下内容都能在界面上真实看到：

- 工作流级变量至少 3 个
- 启动工作流输入变量至少 2 个
- 一个大模型节点
- 一个回复用户节点
- 一个结束节点
- 回复节点成功引用上游变量

### 必须包含的变量建议

- 工作流级变量：
  - `case_id`
  - `case_type`
  - `branch_focus`
- 启动工作流输入变量：
  - `user_query`
  - `user_openid`

### 最简节点顺序

`开始 -> 大模型 -> 回复用户 -> 结束`

### 回传物

- 导出 zip
- 导出后解压目录
- 一张元器界面截图：
  - 工作流级变量面板
  - 启动工作流输入变量面板
  - 回复节点变量引用配置

### 完成标准

- 导入后不是空工作流
- 变量面板能看到
- 回复节点可以引用变量
- 能成功跑出一句固定回复

### 参考文件

- [元器最小回传样例需求（变量面板）](/E:/lawai/docs/元器最小回传样例需求（变量面板）.md)

## 3.2 主线二：得理工具手工配置

### 目标

先把元器里真正要用的法律检索工具建好，后面工作流只需要接这些工具，不再临时返工。

### 建议先建的 3 个工具

- `search_case`
- `search_law`
- `get_law_info`

### 这条线为什么现在就能做

- 它不依赖 workflow zip 修好
- 它本来就必须手工做
- 做完后后续业务工作流才有真实检索底座

### 回传物

- 每个工具的配置截图
- 每个工具一次成功调试结果
- 每个工具的输入输出字段记录

### 参考文件

- [得理检索工具配置清单（元器）](/E:/lawai/docs/得理检索工具配置清单（元器）.md)
- [得理开放平台API接入说明](/E:/lawai/docs/得理开放平台API接入说明.md)

## 3.3 主线三：法律素材库建设

### 目标

把法学同学的知识沉淀成结构化素材，而不是散落聊天记录。

### 第一批必须完成的 2 类模板

- `Demo案件事实包`
- `案由-争点-证据-法条映射表`

### 建议优先做的 4 个案由

- 民间借贷
- 劳动争议
- 离婚纠纷
- 侵权责任

### 法学同学应该交什么

不是长论文，而是结构化表格。

### 完成标准

- 每个案由至少 1 份 Demo 案件事实包
- 每个案由至少 3 条核心争点映射
- 每条争点都能对应证据和法条
