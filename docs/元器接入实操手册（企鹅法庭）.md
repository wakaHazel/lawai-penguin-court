# 元器接入实操手册（企鹅法庭）

> 版本：2026-04-07  
> 项目：企鹅法庭·沉浸式庭审模拟与立法沙盘推演 AI 智能体系统  
> 当前主轴：`案件录入 -> 庭审模拟 -> 对方行为模拟 -> 胜诉率分析 -> 复盘报告`

> 重要说明：  
> 这份文档回答“元器后台里具体怎么搭”。  
> 如果它与 [元器工作流联动契约（企鹅法庭）.md](/E:/lawai/docs/元器工作流联动契约（企鹅法庭）.md) 有冲突，以联动契约为准。

## 1. 先说最终结论

企鹅法庭这套元器工作流，正式联调时不能做成：

- 每个 `workflow_hint` 各调一次元器
- 每条子工作流各跑各的
- 最后再靠人脑拼接

正式模式必须做成：

`每个庭审回合只调用一次 W00_企鹅法庭主控编排`

然后由 `W00` 在元器内部顺序调用：

- `W01_庭审场景生成`
- `W02_法律支持检索`
- `W03_对方行为模拟`
- `W04_结果分析复盘`

只是根据不同阶段，有些子工作流会跳过：

| 当前阶段 | 必跑工作流 |
| --- | --- |
| `prepare` | `W01 -> W02` |
| `investigation` | `W01 -> W02` |
| `evidence` | `W01 -> W02` |
| `debate` | `W01 -> W02 -> W03` |
| `final_statement` | `W01 -> W02 -> W03` |
| `mediation_or_judgment` | `W01 -> W02 -> W04` |
| `report_ready` | `W01 -> W02 -> W04` |

这才叫“环环相扣”。

## 2. 先锁死 4 条原则

### 2.1 主轴不变

元器只服务这一条链路：

`案件录入 -> 庭审模拟 -> 对方行为模拟 -> 胜诉率分析 -> 复盘报告`

不扩成通用多 agent 平台。

### 2.2 阶段推进权在后端

当前项目阶段由后端状态机控制：

- `prepare`
- `investigation`
- `evidence`
- `debate`
- `final_statement`
- `mediation_or_judgment`
- `report_ready`

元器不能：

- 自己决定阶段跳转
- 自己修改 `current_stage`
- 自己决定案件是否结束

### 2.3 第一版不改后端动作校验逻辑

当前后端仍然用本地快照校验：

- `selected_action`
- `available_actions`

所以第一版元器不能成为正式 `available_actions` 的生产者。

第一版只能这样分工：

- 后端：正式 `available_actions`
- 元器：`suggested_actions`

### 2.4 正式模式和调试模式分开

正式模式：

- 一回合一次 `W00`
- `W00` 内部串行调子工作流

调试模式：

- 允许单独调 `W01 / W02 / W03 / W04`
- 只用于排查节点问题
- 不能把调试模式当作正式链路

## 3. 元器里应该建什么

推荐结构：

- 1 个对外主控工作流
- 4 个内部子工作流
- 1 个得理法律检索插件

推荐命名：

| 编号 | 名称 | 作用 |
| --- | --- | --- |
| W00 | `W00_企鹅法庭主控编排` | 每回合唯一入口，顺序驱动子工作流并合并结果 |
| W01 | `W01_庭审场景生成` | 生成当前回合场景文案、角色、建议动作 |
| W02 | `W02_法律支持检索` | 通过得理插件检索法条和案例，并整理摘要 |
| W03 | `W03_对方行为模拟` | 模拟对方本轮动作、抗辩和风险变化 |
| W04 | `W04_结果分析复盘` | 生成胜诉率倾向、风险点、复盘建议和报告草稿 |

插件推荐命名：

- `得理法律检索插件（企鹅法庭）`

## 4. 当前代码和工作流如何对齐

当前后端已经会为不同阶段构造不同的 `workflow_hints`。真实逻辑如下：

### 4.1 固定能力

所有阶段都具备：

- `courtroom_scene_generation`
- `legal_support_retrieval`

### 4.2 对抗阶段能力

当阶段是：

- `debate`
- `final_statement`

额外启用：

- `opponent_behavior_simulation`

### 4.3 结果阶段能力

当阶段是：

- `mediation_or_judgment`
- `report_ready`

额外启用：

- `outcome_analysis_report`

### 4.4 为什么不能一条条独立调

如果每条 `workflow_hint` 单独调一次，会出现两个问题：

1. `W02` 的法律支持结果传不到 `W04`
2. 每条链路各自返回，最后只能靠后端二次拼接，耦合更乱

所以：

- `workflow_hints` 可以继续保留，作为“当前回合需要哪些能力”的结构化提示
- 但正式联调不应继续按 hint 独立发请求
- 正式联调要改成“回合级编排”

## 6. 搭建顺序

### 第 1 步：先建得理插件

先建插件，不先建主控工作流。

推荐建 3 个方法：

1. `search_case`
2. `search_law`
3. `get_law_info`

#### `search_case`

- Method: `POST`
- URL: `https://openapi.delilegal.com/api/qa/v3/search/queryListCase`
- Header:
  - `appid`
  - `secret`

Body 建议：

```json
{
  "pageNo": 1,
  "pageSize": 5,
  "sortField": "correlation",
  "sortOrder": "desc",
  "condition": {
    "keywordArr": ["{{keyword}}"]
  }
}
```

#### `search_law`

- Method: `POST`
- URL: `https://openapi.delilegal.com/api/qa/v3/search/queryListLaw`
- Header:
  - `appid`
  - `secret`

Body 建议：

```json
{
  "pageNo": 1,
  "pageSize": 5,
  "sortField": "correlation",
  "sortOrder": "desc",
  "condition": {
    "keywords": ["{{keyword}}"],
    "fieldName": "semantic"
  }
}
```

#### `get_law_info`

- Method: `GET`
- URL:

```text
https://openapi.delilegal.com/api/qa/v3/search/lawInfo?lawId={{law_id}}&merge=true
```

注意：

- 上面出现的 `{{keyword}}`、`{{law_id}}`、`{{appid}}`、`{{secret}}` 都只是占位表达
- 元器后台的真实配置入口，以实际 UI 为准
- 不要把这里的占位符语法误当成平台唯一写法

### 第 2 步：先建子工作流，再建主控

推荐顺序：

1. `W01_庭审场景生成`
2. `W02_法律支持检索`
3. `W03_对方行为模拟`
4. `W04_结果分析复盘`
5. `W00_企鹅法庭主控编排`

原因：

- 子工作流稳定后，`W00` 只是负责编排
- 如果先做 `W00`，后面每改一条子工作流都要重新回头改总控

## 7. `W01_庭审场景生成` 怎么搭

### 7.1 作用

`W01` 只负责生成：

- `scene_title`
- `scene_text`
- `speaker_role`
- `suggested_actions`
- `branch_focus`
- `next_stage_hint`

它第一版不负责生成系统正式使用的：

- `available_actions`

### 7.2 节点顺序

推荐：

1. 开始节点
2. 代码节点：解析 JSON 字符串数组
3. 大模型节点：生成场景
4. 代码节点：校验输出字段
5. 回复节点

### 7.3 输入变量

- `case_id`
- `current_stage`
- `turn_index`
- `selected_action`
- `next_stage`
- `branch_focus`
- `v_case_type`
- `v_case_title`
- `v_case_summary`
- `v_notes`
- `focus_issues_json`
- `claims_json`
- `missing_evidence_json`
- `opponent_arguments_json`

### 7.4 提示词要求

系统提示词建议：

```text
你是“企鹅法庭”的庭审场景生成器，不是自由聊天助手。

你只负责生成当前回合的场景文案与展示建议。
你不能决定系统正式动作列表，也不能改写庭审阶段。

必须遵守：
1. 不得脱离 current_stage。
2. 不得替后端决定是否跳阶段。
3. suggested_actions 只是建议，不是系统最终可点击动作。
4. 输出必须是 JSON。
5. 文风要像模拟法庭现场，而不是法律论文。
6. 不得编造明显违法、荒谬、失真的司法程序。
7. speaker_role 只能输出这些值之一：
   plaintiff、defendant、applicant、respondent、agent、witness、judge、other。

输出 JSON：
{
  "scene_title": "",
  "scene_text": "",
  "speaker_role": "",
  "suggested_actions": [],
  "branch_focus": "",
  "next_stage_hint": ""
}
```

### 7.5 输出结构

```json
{
  "workflow_key": "courtroom_scene_generation",
  "status": "ok",
  "payload": {
    "scene_title": "",
    "scene_text": "",
    "speaker_role": "",
    "suggested_actions": [],
    "branch_focus": "",
    "next_stage_hint": ""
  }
}
```

## 8. `W02_法律支持检索` 怎么搭

### 8.1 作用

`W02` 负责：

- 调得理插件查法条
- 调得理插件查案例
- 生成可读的法律支持摘要
- 把结果整理成能继续传给 `W03 / W04` 的结构化产物

### 8.2 节点顺序

推荐：

1. 开始节点
2. 代码节点：解析 `focus_issues_json` 和 `fact_keywords_json`
3. 插件节点：`search_law`
4. 插件节点：`search_case`
5. 可选插件节点：`get_law_info`
6. 大模型节点：整理摘要
7. 回复节点

### 8.3 输入变量

- `case_id`
- `case_type`
- `focus_issues_json`
- `fact_keywords_json`

### 8.4 代码节点要做什么

只做 3 件事：

1. 把 JSON 字符串数组转回数组
2. 生成 1 到 3 个检索关键词
3. 空数组时给兜底关键词

### 8.5 提示词要求

```text
你是“企鹅法庭”的法律支持整理器。

你只负责整理得理插件返回的结果，不得凭空捏造法条。

必须遵守：
1. 输出必须是 JSON。
2. 每条法律支持尽量对应争议焦点。
3. 要指出当前事实和证据还不足的地方。
4. 输出结果要能继续传给 W03 和 W04。

输出 JSON：
{
  "legal_support_summary": "",
  "recommended_laws": [],
  "recommended_cases": [],
  "issue_mapping": [],
  "missing_points": []
}
```

### 8.6 输出结构

```json
{
  "workflow_key": "legal_support_retrieval",
  "status": "ok",
  "payload": {
    "legal_support_summary": "",
    "recommended_laws": [],
    "recommended_cases": [],
    "issue_mapping": [],
    "missing_points": []
  }
}
```

## 11. `W00_企鹅法庭主控编排` 怎么搭

### 11.1 作用

`W00` 不负责创造法律内容，它负责：

- 接收回合级变量
- 判断当前阶段
- 顺序调用子工作流
- 合并结果
- 返回统一 JSON

### 11.2 起始变量

建议最少有：

- `case_id`
- `simulation_id`
- `current_stage`
- `turn_index`
- `selected_action`
- `branch_focus`
- `case_type`
- `v_case_type`
- `v_case_title`
- `v_case_summary`
- `v_notes`
- `focus_issues_json`
- `claims_json`
- `missing_evidence_json`
- `fact_keywords_json`
- `opponent_arguments_json`
- `opponent_role`
- `opponent_name`
- `likely_arguments_json`
- `likely_evidence_json`
- `likely_strategies_json`

### 11.3 节点顺序

正式模式推荐：

1. 开始节点
2. 代码节点：标准化输入
3. 子工作流节点：`W01`
4. 子工作流节点：`W02`
5. 分支节点：
   - 如果阶段是 `debate / final_statement`，调用 `W03`
   - 如果阶段是 `mediation_or_judgment / report_ready`，调用 `W04`
6. 代码节点：统一合并结果
7. 回复节点

### 11.4 `W00` 统一回包结构

建议最终回：

```json
{
  "status": "ok",
  "stage": "",
  "scene": {},
  "legal_support": {},
  "opponent": {},
  "analysis": {},
  "degraded_flags": []
}
```

其中：

- `scene` 来自 `W01`
- `legal_support` 来自 `W02`
- `opponent` 来自 `W03`
- `analysis` 来自 `W04`

### 14.3 `W03` 单测

验证：

- 能输出 `opponent_action`
- 能输出 `opponent_argument`
- `opponent_role` 不乱写

### 14.4 `W00` 正式链路测试

验证：

- `prepare / investigation / evidence` 只跑 `W01 -> W02`
- `debate / final_statement` 跑 `W01 -> W02 -> W03`
- `mediation_or_judgment / report_ready` 跑 `W01 -> W02 -> W04`
- 上游结果能进下游，不断链

## 15. 最容易出错的点

1. 让 `W01` 直接生成正式 `available_actions`
2. 让 `W04` 不消费 `W02` 结果就开始分析
3. 把数组直接传给元器，不先转 JSON 字符串
4. 让模型输出自然语言角色，而不是后端枚举值
5. 把插件占位写法误当成元器后台固定语法

## 16. 一句话结论

如果你要的是一个真的不会断链、而且后面不容易返工的元器框架，那就按这条做：

`每回合只调一次 W00 -> W00 顺序调 W01/W02/W03/W04 -> 后端适配、缓存、合并 -> 前端展示`

不要再按“每个能力各发一次请求”理解正式工作流了。
