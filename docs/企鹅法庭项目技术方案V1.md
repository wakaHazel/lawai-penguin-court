## 1. 赛题硬约束

根据你补充的 D06 正式题目说明，本项目方案必须先满足以下硬约束，再谈技术实现：

- 项目类型必须属于 `法律 AI 应用创新与实践` 赛题范畴。
- 项目形态可选 `智能体 / 工具应用 / 系统解决方案`，其中本项目应明确归类为：
  - `面向法律场景的系统解决方案`
  - 具体落点：`沉浸式庭审模拟与法律辅助分析系统`
- 对外提交时，必须明确说明本项目 `基于腾讯系 AI 工具平台完成设计与开发`。
- 赛题要求中明确列出的开发工具白名单只有 6 个：
  - `腾讯开悟比赛平台`
  - `CodeBuddy`
  - `腾讯元器`
  - `腾讯元宝`
  - `小理AI`
  - `得理开放平台API`
- 因此在比赛材料、答辩表述、项目说明书、AI工具使用说明中，必须把上述 6 项写成主工具链，不应再写成“可替换”“可自由选配”的口径。
- 区域初赛与后续决赛材料中，`AI工具使用说明（工作流与prompt）` 都是明确交付项，不能只写产品功能，必须把元器工作流和 Prompt 设计单独成章。

## 2. 方案结论

本项目不应采用“开放式多 agent 平台”作为主框架，而应采用一套面向赛事交付的受控智能体系统架构：

`文游前端交互层 + 元器工作流编排层 + 后端控制与持久化层 + 3 个核心智能体 + 法律检索工具层 + 报告生成层`

推荐实现假设分两层表达：

- `赛题主工具链`
  - `腾讯开悟比赛平台`
  - `CodeBuddy`
  - `腾讯元器`
  - `腾讯元宝`
  - `小理AI`
  - `得理开放平台API`
- `工程落地实现假设`
  - 前端：`React + Vite + TypeScript`
  - 后端：`Python 3.11 + FastAPI`
  - 数据层：`SQLite 或同类轻量本地库 + JSON 配置`
  - 报告层：`Markdown/HTML 模板生成`

这里必须区分清楚：

- 前一层是 `赛题要求必须出现并重点说明的官方工具链`
- 后一层是 `为完成作品部署与演示所采用的常规工程实现`

对外答辩时，主叙事必须始终放在前一层。

这套方案的核心优势是：

- 与 `.docx` 的产品叙事和技术叙事一致
- 与 D06 帮助指引中“工作流推荐用元器、代码平台推荐用 CodeBuddy”的口径一致
- 符合 15 天赛题交付节奏
- 便于法学同学和导师做法律内容校验
- 便于现场演示和答辩，不会因开放式 agent 行为失控

## 3. 为什么不用“通用多 agent 平台”做主轴

原始 `.docx` 已经限定了项目主线：

- MVP 核心是 `庭审模拟系统`
- 加分项是 `立法沙盘推演`
- 腾讯元素必须深度绑定
- 智能体应围绕庭审交互、对方行为模拟、胜诉率分析展开
- 赛事展示强调 `文游式多分支体验` 和 `法律专业性`

因此项目的核心挑战不是“如何让很多 agent 自主协作”，而是：

- 如何严格控制庭审阶段顺序
- 如何生成高沉浸感但不失真的剧情文本
- 如何让法律内容可解释、可校验、可追溯
- 如何在比赛环境下稳定输出

结论：

- 可以借鉴图式编排、工具抽象、结构化输出等思想
- 不应把 `CrewAI`、`AutoGen`、`LangGraph` 或其他通用框架当作项目主叙事
- 对外方案必须围绕 `腾讯元器/元宝 + CodeBuddy + 腾讯开悟平台 + 得理 API + 企鹅 IP + 文游式庭审体验`

## 3.1 腾讯工具链主线

依据正式赛题说明与 D06 帮助指引，本项目的官方工具链必须明确如下：

- `腾讯元器`
  - 作为智能体工作流编排的优先载体
  - 承载案例检索、法规检索、意图识别、变量聚合、多分支聚合等工作流节点
- `腾讯元宝`
  - 用于 prompt 设计、场景文本调优、报告文本润色
- `CodeBuddy`
  - 作为推荐代码平台，用于前端页面和工程开发协同
- `腾讯开悟平台`
  - 作为赛道平台适配、学习资料获取、链路打通、提交和展示链路的一部分
- `得理开放平台 API`
  - 提供案例检索、法规检索两个核心原子能力
  - 在元器平台中通过插件接口提供法律数据服务
  - 接入细节见 [得理开放平台 API 接入说明](./得理开放平台API接入说明.md)
- `小理 AI`
  - 作为参赛福利资源和法律实务应用案例参考来源
  - 主要承担案例理解、法规理解、答辩材料润色的备选支持

## 3.2 元器与后端的分工边界

依据 D06 帮助指引，“智能体工作流编排推荐用元器”，因此本项目不应把元器降级成单纯模型接口。

推荐分工如下：

- `元器负责`
  - 智能体工作流编排
  - 案例检索和法规检索的工具节点调用
  - 意图识别
  - 工作流变量处理、变量聚合、回复输出
- `后端负责`
  - 业务状态持久化
  - 案件会话管理
  - 庭审阶段控制与规则校验
  - 本地规则库兜底
  - 报告归档、示例包生成、比赛包装

也就是说，元器是工作流主载体，后端是业务控制与稳定性保障层，两者协同，而不是互相替代。

## 4. 项目目标与范围

## 4.1 MVP 必做

按 `.docx`，MVP 应确保以下闭环：

1. 案件信息可视化录入
2. 文游式全流程庭审模拟
3. 多结局胜诉率分析
4. 《庭审模拟复盘报告》生成
5. 《庭前备战清单》生成
6. 顶层案件类型支持 `民事 / 刑事 / 行政`
7. 优先完整覆盖 4 类高频案件：`民间借贷 / 劳动争议 / 离婚纠纷 / 侵权责任`
8. 3 组输入输出示例包
9. 可运行 demo 与提交材料

## 4.2 加分项

立法沙盘推演系统为加分拓展模块，可在 MVP 稳定后以轻量版实现：

- 方案录入
- 条款变量调整
- 沙盘推演
- 评估报告

## 4.3 当前明确不做

为保证项目聚焦，不建议在首版中加入：

- 复杂用户体系
- 云端协作与多租户
- 开放式通用法律问答
- 高自由度 agent 自主协商框架

## 5.2 选型结论

项目采用 `单仓 + 双应用 + 元器工作流协同 + 后端模块化单体`。

- 前端是单独的 Web 应用
- 后端是单独的 API 应用
- 后端内部采用模块化单体，不拆微服务

原因：

- 团队规模小，微服务会带来额外部署和联调负担
- D06 帮助指引已明确推荐用元器做工作流编排，因此后端不必再自建重量级编排平台
- MVP 周期短，优先保证单机可跑、可测、可打包
- 法律流程和报告生成需要共享上下文，单体更稳定
- 若后续科创项目扩展，再从模块边界中抽服务

## 6. 3 个核心智能体设计

## 6.1 庭审流程交互智能体

职责：

- 生成当前庭审阶段场景文本
- 决定当前出场角色
- 返回用户可点击的策略分支
- 将专业内容包装成文游式表达

输入：

- 案件结构化信息
- 当前阶段
- 上一轮用户选择
- 当前争议焦点

输出：

- `scene_title`
- `scene_text`
- `speaker_roles`
- `branch_options`
- `next_stage_hint`

约束：

- 不允许跳过法定阶段
- 不允许凭空发明核心事实
- 所有分支必须回到状态机可控范围

## 6.2 对方行为模拟智能体

职责：

- 模拟对方律师、对方当事人、证人等行为
- 生成突袭证据、抗辩意见、证言变化
- 增强庭审真实感和不确定性

输入：

- 对方画像
- 案件类型
- 当前阶段
- 对方可用证据池
- 已知策略空间

输出：

- `opponent_action`
- `evidence_event`
- `argument_text`
- `risk_delta`

约束：

- 仅能在已定义的 surprise pool 内产生新事件
- 不得输出违反案件设定和法律逻辑的行为

## 6.3 胜诉率分析智能体

职责：

- 基于全流程事件和法律依据生成评分
- 输出庭审复盘报告
- 输出庭前备战建议

输入：

- 案件信息
- 全量事件时间线
- 法律检索结果
- 证据与分支结果

输出：

- `claim_support_rate`
- `evidence_acceptance_rate`
- `focus_issue_win_rate`
- `risk_points`
- `advice_items`
- `report_markdown`

约束：

- 评分必须分维度解释
- 不能只给笼统结论
- 报告必须包含法律依据来源类别

## 7. 庭审流程编排内核

## 7.1 固定阶段

建议严格固定为：

- `PREPARE`
- `INVESTIGATION`
- `EVIDENCE`
- `DEBATE`
- `FINAL_STATEMENT`
- `MEDIATION_OR_JUDGMENT`
- `REPORT_READY`

## 7.2 每阶段包含

- 阶段名
- 阶段目标
- 可出场角色
- 可触发事件
- 用户可选策略
- 允许调用的智能体
- 风险分更新规则
- 转移到下一阶段的条件

## 7.3 为什么必须状态机优先

因为 `.docx` 的要求里有两个不能妥协的点：

- “严格按照法定庭审流程”
- “关键节点设置分支选择”

这说明：

- 顺序必须是确定的
- 分支必须是受控的
- 模型只在节点内部生成内容，不能接管流程控制

## 8. 法律工具层设计

## 8.1 外部法律能力

主链路首选：

- 得理法律 API
- 小理 AI 作为辅助分析与备选能力

得理 API 的具体端点、认证方式、参数类型约束和错误处理要求，统一按 [得理开放平台 API 接入说明](./得理开放平台API接入说明.md) 执行。

用途：

- 法条检索
- 类案匹配
- 法律文书生成支撑

## 8.2 本地规则库

必须建立本地规则库作为兜底，内容包括：

- 庭审流程规则
- 高频案件争议焦点
- 常见抗辩方向
- 证据采信规则
- 胜诉率因子及权重

## 8.3 模板库

必须建立模板库：

- 角色模板
- 剧情模板
- 突袭证据模板
- 报告模板
- 立法评估模板

原因：

- 保证输出稳定
- 降低模型幻觉风险
- 比赛现场出现 API 波动时仍能演示主链路

## 9. 数据层设计

数据库采用 SQLite 即可，实体建议包括：

- `cases`
- `case_profiles`
- `evidence_items`
- `simulation_turns`
- `risk_profiles`
- `report_artifacts`
- `demo_cases`

注：

- `SQLite` 是推荐实现假设，目的是满足比赛周期内的单机稳定演示
- 如果团队已有更熟悉的轻量存储方案，可替换，但不应引入重型分布式依赖

另保留 JSON 文件：

- `trial_flow.json`
- `case_type_rules/*.json`
- `surprise_events/*.json`
- `report_templates/*.md`
- `demo_cases/*.json`

## 10. 前端方案

前端重点不是传统管理后台，而是“文游化庭审体验”。

必须具备的页面：

- 案件录入页
- 庭审模拟页
- 报告页
- 立法沙盘页

核心交互组件建议：

- `StageProgress`
- `PenguinRolePanel`
- `SceneNarrative`
- `ChoiceActionBar`
- `EvidenceCard`
- `RiskSummary`
- `ReportViewer`

视觉和交互重点：

- 当前阶段明确可见
- 角色出场有强识别度
- 分支按钮必须清晰
- 剧情文本有“CG 式”表达感
- 报告页可直接用于答辩展示

## 15. 风险控制

## 15.1 技术延期

处理策略：

- 先冻结 MVP 边界
- 立法沙盘只保留轻量入口和预置案例
- 绝不在首版引入额外复杂平台依赖
- 顶层案件类型保留 `民事 / 刑事 / 行政`，MVP 完整实现 4 个高频子类

## 15.2 法律内容错误

处理策略：

- 规则库与模板必须以现行法律法规、司法解释、指导案例为依据
- 规则库与模板必须先过法学双人校验
- 高校法学院导师审核前，不得冻结最终报告模板和规则库
- 检索结果只做辅助，最终表达按规则库和模板约束
- 报告明确区分“法条依据”“类案参考”“模拟建议”

## 15.3 模型输出不稳定

处理策略：

- 所有智能体输出必须走结构化 schema 校验
- 失败重试一次
- 再失败切模板兜底
- 元器工作流中使用变量聚合避免多分支输出 `null`
- 数组参数统一通过代码节点或后端转换，避免 `ArrayString` 类型错误

## 15.4 比赛现场环境不稳

处理策略：

- demo 案例本地缓存
- 法律规则本地化
- 预置剧情分支
- 保留小理 AI 作为辅助分析和应急备选资源
- 保证离线或弱网环境下仍能演示关键流程

## 16. 最终建议

这不是一个“为了用多 agent 而用多 agent”的项目。  
它应该被定义为一个：

`面向法律教育与诉讼准备场景的，企鹅 IP 驱动的文游式受控智能体系统`

因此正确的技术方向是：

- 用 `元器工作流` 承载智能体编排
- 用 `后端控制层` 管理状态、规则、持久化和比赛包装
- 用 `3 个专职智能体` 生成内容
- 用 `法律工具层` 提供依据
- 用 `前端文游体验` 放大创新点

如果后续作为长期科创项目继续扩展，再在这个底座上增加：

- 立法沙盘深度模块
- 数据分析后台
- 多角色训练场
- 高校教学模式

但首版绝不能把主线从“庭审模拟产品”做成“多 agent 平台产品”。
