# 企鹅法庭 5人分工与并行推进清单

> 版本：2026-04-06
> 适用范围：`企鹅法庭·沉浸式庭审模拟与立法沙盘推演AI智能体系统` MVP 阶段
> 对齐依据：`法律ai服务大赛.docx`、`服创赛D06赛道帮助指引.docx`

## 1. 先说结论

法律同学能做的事情，我可以替你们先做掉相当大一部分，但不能完全替代法律同学。

更准确地说：

- 我可以承担 `60% - 80%` 的结构化产出工作。
- 我不能替代法律同学做 `最终法律判断、争议取舍、法条适用把关、答辩背书`。

最合适的合作方式不是“AI 替代法学生”，而是：

- `我负责高速起草、整理、归纳、结构化、生成初稿`
- `法律同学负责审校、纠偏、确认、背书`

这样效率最高，也最适合你这个科创比赛项目。

## 2. 我能直接替法律同学做的事

以下这些，我现在就能做，而且通常能做得很快：

- 把赛题要求拆成系统模块、页面、功能清单。
- 把案件流程拆成 `案件录入 -> 庭前准备 -> 庭审模拟 -> 胜诉率分析 -> 复盘报告`。
- 生成 `案件字段表`、`证据字段表`、`争点字段表`、`胜诉率评分字段表`。
- 按民事/刑事/行政三个大类，先整理 MVP 应优先支持的案件类型。
- 为 `民间借贷 / 劳动争议 / 离婚纠纷 / 侵权责任` 生成标准化案例模板。
- 生成庭审模拟中的角色提示词初稿：
  - 庭审流程交互智能体
  - 对方行为模拟智能体
  - 胜诉率分析智能体
- 生成法律检索流程初稿，包括：
  - 该查什么法条
  - 该查什么案例
  - 证据不足时如何提示用户补充
- 生成 `庭前备战清单` 初稿。
- 生成演示用 `demo 案件包` 初稿。
- 生成 `输入输出示例`、`作品说明文件`、`使用说明`、`答辩问答库` 初稿。
- 帮你把法律材料整理成适合工程实现的规则表，而不是散乱文字。
- 帮你把法律同学写的内容转成系统 prompt、评分规则、流程节点说明。

## 3. 法律同学必须亲自做的事

下面这些，不建议完全交给我，必须由法律同学最终确认：

- 确认法条引用是否准确，是否存在理解偏差。
- 确认案例类型划分是否合理，是否符合真实司法实践。
- 确认争点抽取是否抓住核心，而不是表面描述。
- 确认证据三性、证明力、举证责任分配是否表达准确。
- 确认对方代理人/法官/当事人口径是否符合真实庭审语境。
- 确认胜诉率分析中的法律推理是否存在“说得像，但法律上不严谨”的问题。
- 确认比赛答辩中涉及法律效果、社会价值、合规风险的表述。
- 最终对外提交材料时，承担法律内容的人工审校和背书。

一句话说清楚：

- `我可以生成初稿`
- `她们必须做终稿把关`

## 4. 最推荐的五人分工

按你们原始项目设定，五个人最稳的分法如下。

### 4.1 产品&项目负责人

负责：

- 锁定项目边界，不让项目跑偏成“泛法务聊天机器人”。
- 控制比赛节奏、文档版本、提交材料。
- 统一协调法学同学和技术同学的接口。
- 决定 MVP 只做哪些案由，不贪多。
- 统筹答辩 PPT、演示视频、命名规范、提交清单。

并行可做：

- 我帮你写方案、计划、PPT 大纲、答辩问答库。
- 你负责拍板、取舍、对外呈现。

### 4.2 全栈开发工程师

负责：

- 前端页面搭建。
- 后端接口、数据结构、状态持久化。
- 接入元器工作流和后端控制层。
- 接入 DeliLegal API。
- 打通 demo 流程。

并行可做：

- 我可以继续帮你写接口设计、字段定义、流程说明、后端骨架方案。
- 她/他可以直接开始搭 `apps/web` 和 `apps/api`。

### 4.3 技术顾问

负责：

- 技术路线把关。
- 元器、CodeBuddy、开悟平台等工具链选型与风险控制。
- 架构审查。
- 关键节点验收。

并行可做：

- 我先把架构方案、接口方案、工作流方案写出来。
- 技术顾问只需要做高价值评审，不必从零起草。

### 4.4 法学同学 A

最适合负责 `规则与争点侧`：

- 整理 4 个 MVP 案由的核心争点。
- 整理每类案由的证据清单。
- 整理每类案由的常见抗辩点。
- 整理每类案由对应的法条、司法解释、裁判要点。
- 审校庭审流程中的法律术语是否准确。

她的直接交付物建议是：

- `案由-争点-证据-法条映射表`
- `案由-常见抗辩与反制表`
- `胜诉率分析人工校正规则表`

### 4.5 法学同学 B

最适合负责 `案例与演示侧`：

- 按 MVP 案由各写 1 个高质量 demo 案件。
- 补充案件事实、时间线、证据目录、争议焦点。
- 审校 AI 生成的庭审对话是否像“真实庭审”。
- 协助准备答辩中的法律价值表达。
- 协助准备作品说明中的应用场景与社会意义。

她的直接交付物建议是：

- `4个 demo 案件事实包`
- `案件时间线与证据目录表`
- `庭审模拟对白审校意见表`

## 5. 法律同学和我怎么配合效率最高

最省时间的方式不是让法律同学写长篇文字，而是让她们按表格给我素材。

最推荐她们给我的输入格式如下：

### 5.1 争点映射表

字段建议：

- `案由`
- `核心争点`
- `原告主张`
- `被告抗辩`
- `关键法条`
- `关键证据`
- `常见败诉点`

### 5.2 Demo 案件事实包

字段建议：

- `案由`
- `当事人身份`
- `案件背景`
- `关键事实时间线`
- `已有证据`
- `缺失证据`
- `对方可能说法`
- `希望系统输出什么`

### 5.3 审校意见表

字段建议：

- `原文`
- `问题类型`
- `为什么不严谨`
- `建议改法`
- `是否必须修改`

只要她们按这个格式给，我就能很快把内容转成：

- 产品文档
- 页面字段
- prompt
- 工作流节点
- 报告模板
- 演示脚本

## 6. 今天就能并行开的任务

你们现在完全可以并行，不需要互相等。

### 你这边可以立刻开始

- 继续让我补全技术方案、团队分工、字段表、工作流设计。
- 确定 MVP 只保留 4 个高频案由。
- 确定元器作为主工作流编排载体。
- 确定 DeliLegal API 作为法条/案例检索主工具。

### 全栈同学可以立刻开始

- 搭前端页面骨架：
  - 案件录入页
  - 庭审模拟页
  - 胜诉率分析页
  - 庭前准备清单页
- 搭后端项目骨架：
  - case
  - simulation
  - report
  - health

### 法学同学 A 可以立刻开始

- 先做 `民间借贷`、`劳动争议` 两个案由的争点与证据表。
- 不用写长文，直接写结构化表格。

### 法学同学 B 可以立刻开始

- 先写 2 个高质量 demo 案件：
  - 民间借贷
  - 离婚纠纷
- 每个案件至少给出：
  - 基本事实
  - 时间线
  - 证据目录
  - 对方可能抗辩

### 产品负责人可以立刻开始

- 收敛比赛交付清单。
- 准备答辩故事线：
  - 痛点
  - 产品
  - 技术
  - 创新
  - 演示
  - 社会价值

## 7. 三天内最现实的推进节奏

### Day 1

- 我输出：
  - 技术方案
  - 分工清单
  - 字段表
  - 工作流草案
- 技术侧输出：
  - 前后端空项目骨架
- 法学侧输出：
  - 2 个案由争点表
  - 2 个 demo 案件事实包

### Day 2

- 我输出：
  - prompt 初稿
  - 报告模板初稿
  - 庭前备战清单初稿
- 技术侧输出：
  - 案件录入页
  - 模拟流程接口
- 法学侧输出：
  - 其余 2 个案由争点表
  - 全部 demo 案件补完

### Day 3

- 我输出：
  - 答辩问答库初稿
  - 作品说明初稿
  - 输入输出示例初稿
- 技术侧输出：
  - 可跑通 demo
- 法学侧输出：
  - 对 AI 文稿进行一轮红笔审校

## 8. 最后给你的直接判断

如果你的问题是：

`法律同学能做的，你能做吗？`

我的直接回答是：

- `能做大部分前置产出`
- `不能替代最终法律把关`

如果你的目标是提高速度，那我非常适合当：

- `法律材料整理器`
- `规则表生成器`
- `庭审脚本起草器`
- `答辩材料加速器`

但如果你的目标是完全不要法律同学，那不稳，也不适合比赛场景。

最优解是：

- `我负责把 80% 的草稿和结构先打出来`
- `法律同学负责把最后 20% 的专业准确性钉死`

这样最快，也最安全。
