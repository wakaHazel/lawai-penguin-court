# 元器实施边界与约束（企鹅法庭）

## 1. 这份文档解决什么问题

当前最容易混淆的是三件事：

1. `导入包` 负责的是 `W00-W04` 业务工作流骨架，不等于全部元器配置。
2. 得理开放平台的 `search_case / search_law / get_law_info` 属于 `工具层`，需要在元器里单独配置。
3. 元器导出包对 `工作流级变量 / 启动工作流输入变量` 的保留并不稳定，不能假设 zip 导入后 UI 一定自动补全。

如果这三层不分开，后面就会反复出现两类问题：

- 以为导入 zip 后就会自带得理检索工具，结果工作流里找不到工具可选。
- 以为 json 里写了参数，元器 UI 就一定会显示“工作流级变量 / 启动工作流输入变量”，结果导入后还是空的。

## 2. D06 指引里已经确认的硬约束

以下内容来自 `E:\lawai\tmp\docs\d06_guide_paragraphs.txt` 抽取结果，以及整理后的元器手册：

- 得理开放平台这次比赛只明确提供两类核心检索能力：`案例检索`、`法规检索`。
- 法规检索列表本身不包含完整法条正文，若要拿到法规详情，需要继续调用 `lawInfo`。
- `keywordArr` 必须是字符串数组 `[str]`，如果传成 `str`，就会出现 `ArrayString` 错误。
- 多分支直接进回复节点时，容易因为部分分支没有值而输出 `null`，官方示例明确建议先做 `变量聚合`。
- `结束节点` 不负责面向用户的文本回复；如果要在单工作流里直接给人看，需要额外接 `回复节点`。
- 元器工作流编排是推荐方案，但得理开放平台 API 也允许通过第三方代码框架调用。
- `appid` 和 `secret` 在这次赛题里是通过请求头一起传，不要把这套双 Header 逻辑偷换成单一 `ApiKey` 假设。

## 3. 必须分清的三层结构

### 第一层：工具层

这层是元器里的 `插件 / 工具` 配置，不在当前导入包里。

必须手动建的工具有 3 个：

1. `search_case`
2. `search_law`
3. `get_law_info`

这 3 个工具配置好以后，业务工作流或者主智能体才能调用它们。

### 第二层：业务工作流层

这层才是当前导入包负责的内容。

当前项目工作流骨架是：

1. `W00_企鹅法庭主控编排`
2. `W01_庭审场景生成`
3. `W02_法律支持检索`
4. `W03_对方行为模拟`
5. `W04_结果分析复盘`

这几个工作流的职责是：

- `W00` 负责总编排和子工作流引用。
- `W01-W04` 负责阶段化业务输出。
- 子工作流正式联调时，必须通过 `结束节点` 给 `W00` 回传结构化结果。

### 第三层：元器 UI 手工配置层

这一层最容易被忽略，但实际必须保留手工校对动作。

重点检查：

- `开始节点 -> API 参数 -> 管理`
- `工作流级变量`
- `启动工作流输入变量`
- `结束节点输出变量`
- `回复节点引用变量`

结论很直接：

- `json 里存在参数` 不等于 `元器 UI 一定会显示完整变量面板`。
- 如果导入后 UI 仍然空白，要以元器界面为准，手动补齐。

## 4. 当前导入包的真实边界

当前生成器和导入包已经完成的部分：

- 5 个工作流都能生成独立 workflow json。
- `StartNodeData.WorkflowParams` 已写入默认参数。
- 首个代码节点已经从 `API.xxx` 和 `SYS.RewriteQuery` 接输入。
- 第二个代码节点已经统一产出 `answer_text`。
- `ANSWER` 节点已改成引用 `{{answer_text}}`。
- `END` 节点已经绑定上游输出，而不是空结束。

当前导入包没有自动解决的部分：

- 不会自动替你在元器里创建 `search_case / search_law / get_law_info` 工具。
- 不能保证导入后元器 UI 一定自动显示完整的 `工作流级变量 / 启动工作流输入变量` 面板。
- 不能替代手工把 `appid / secret` 真正绑到工具 Header。

所以，正确理解应该是：

- `导入包` 是业务工作流骨架。
- `工具配置` 是手工接入步骤。
- `变量面板` 是导入后必须人工核对的界面配置。

## 5. W02 的正确定位

`W02_法律支持检索` 不是“空口生成法律意见”的普通大模型节点。

它的正确定位应该是：

1. 根据案件焦点生成检索关键词。
2. 调用 `search_law` 和 `search_case`。
3. 如有必要，对命中的法规继续调用 `get_law_info`。
4. 对检索结果做结构化总结。
5. 通过聚合或封装节点统一输出，避免 `null` 直出。

如果后续在元器里搭正式版 `W02`，优先采用这个链路，不要退回到“只有大模型、没有检索”的空转版本。

## 6. 为什么现在不继续硬猜变量表

已经确认的问题是：

- 当前导出机制对 `变量.xlsx`、`参数.xlsx`、`StartNodeData.WorkflowParams`、元器 UI 面板之间的映射不是一一稳定的。
- 真实用户导出的样例包里，也出现了 `WorkflowParams` 为空、`参数.xlsx` 和 `变量.xlsx` 近乎空表的情况。

所以现在最稳的策略不是继续盲猜，而是：

1. 用当前导入包保住业务工作流骨架。
2. 用手工工具配置保住得理检索能力。
3. 下一轮拿一个“已经在元器界面手动配好变量面板”的最小导出样例，再做差异比对。

## 7. 明天继续时的固定顺序

不要跳步，按下面顺序继续：

1. 在元器里先手工建 `search_case / search_law / get_law_info`。
2. 单独测试这 3 个工具能返回结果。
3. 再导入 `W00-W04` 工作流包。
4. 导入后逐个检查 `开始节点参数`、`结束节点输出`、`回复节点引用`。
5. 先联调 `W02`，确保法律支持链能吃到真实检索结果。
6. 再回头联调 `W00` 主控编排。

## 8. 一句话结论

`zip 导入包` 不是“整套元器应用的一键还原包”，而是“企鹅法庭业务工作流骨架包”。

真正可落地的完整实现必须同时具备：

- 手工配置好的得理检索工具
- 导入后的业务工作流
- 元器界面内核对完成的变量面板
