# Penguin Court MVP Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a competition-ready MVP for `企鹅法庭·沉浸式庭审模拟与立法沙盘推演 AI 智能体系统`, centered on the courtroom simulation loop defined in the original `.docx`.

**Architecture:** Use `腾讯元器` as the preferred workflow orchestration carrier for the legal-agent flow, with a modular-monolith backend handling state persistence, guardrails, legal-rule fallback, and submission packaging. Pair it with a lightweight web frontend optimized for story-style courtroom interaction and keep the whole system aligned with the D06 official tool whitelist rather than turning it into a generic multi-agent platform.

**Official Tool Whitelist:** The competition explicitly limits the core development toolchain to:

- 腾讯开悟比赛平台
- CodeBuddy
- 腾讯元器
- 腾讯元宝
- 小理 AI
- 得理开放平台 API

**Engineering Assumptions:** React, Vite, TypeScript, Tailwind CSS, Zustand, Python 3.11, FastAPI, Pydantic, SQLModel/SQLAlchemy, SQLite or equivalent lightweight local storage, Markdown/HTML templates.

**Presentation Rule:** In all outward-facing competition materials, the six official tools above must be presented as the primary development toolchain. The engineering assumptions are supporting implementation details, not the competition-facing innovation tool stack.

---

## Scope Lock

This plan intentionally covers only the MVP path required by the original `.docx`:

- 案件录入
- 庭审模拟
- 对方行为模拟
- 胜诉率分析
- 复盘报告
- 庭前备战清单
- 顶层案件类型支持：民事 / 刑事 / 行政
- MVP 优先完整覆盖：民间借贷 / 劳动争议 / 离婚纠纷 / 侵权责任
- 3 组 demo 输入输出样例

The `立法沙盘` module is included only as a shell and stretch task after the MVP loop is stable.

- `apps/api/routes`: HTTP entry points only.
- `apps/api/orchestrators`: Trial state machine and scene progression.
- `apps/api/agents`: Three core agent adapters and response validators.

- `apps/api/templates`: Report and fallback narrative templates.
- `data/demo_cases`: Stable competition demo inputs.

## Implementation Strategy

Build in this order:

1. 固定案件 schema 与案件类型范围
2. 固定庭审状态机
3. 打通首条可跑链路：`案件录入 -> 启动模拟 -> 第一轮剧情`
4. 再补齐多轮推进
5. 再补齐评分与报告
6. 最后补强 demo、说明书、材料导出

This ordering matches the competition risk profile: first make the showable loop work, then deepen it.
