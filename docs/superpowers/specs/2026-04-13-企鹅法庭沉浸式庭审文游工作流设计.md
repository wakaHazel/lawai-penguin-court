# 企鹅法庭沉浸式庭审文游工作流设计

日期：2026-04-13  
状态：已定版，待评审  
适用范围：企鹅法庭项目第一版核心工作流设计  
设计目标：把“严格依照民事庭审流程的沉浸式文游庭审模拟”确定为系统核心主轴，并为后续后端实现、前端渲染、元器演示映射提供统一规格。

## 1. 设计结论

本项目不是通用法律问答系统，也不是多功能法律工具箱的松散拼接。

第一版的产品核心必须固定为：

`严格依照民事诉讼法庭审流程运行的沉浸式文游式庭审模拟模块`

其余模块的定位全部围绕该主轴服务：

- 案件录入：负责开局建档与标准化案件画像
- 对方推演：负责庭审节点中的敌手策略与突袭内容生成
- 胜诉率分析：负责在关键节点后给出趋势反馈
- 复盘报告：负责生成本轮推演的结算与改进建议
- 历史案件：负责保存多轮推演、关键存档点与回放入口

元器不承担核心推演逻辑，只承担演示层、展示层和轻量接入层。

## 2. 产品定位与主轴

### 2.1 产品定位

企鹅法庭是一个法律程序驱动的互动叙事系统。  
它的目标不是替代法官作出真实裁判，而是让用户以文字冒险游戏的方式，体验一轮高还原度的民事庭审推演，并从中获得策略反馈。

### 2.2 固定主轴

第一版固定六大庭审阶段，不得随意跳过或重排：

1. 开庭准备
2. 法庭调查
3. 举证质证
4. 法庭辩论
5. 最后陈述
6. 调解/宣判

### 2.3 第一版非目标

以下内容不属于第一版必须实现的范围：

- 自由聊天式无限庭审
- 无穷展开的剧情树
- 完全开放的玩家自然语言输入作为主交互方式
- 真实裁判结论替代
- 依赖元器才能完成的主链逻辑

## 3. 设计原则

### 3.1 法律流程硬约束

法律程序是硬骨架，不能由模型自由决定。  
每个节点的位置、阶段归属和可选策略都必须受规则控制。

### 3.2 文游表现软渲染

CG式文本、企鹅角色动作、法庭氛围和情绪张力由渲染器表达，但渲染器不能凭空发明新事实。

### 3.3 分支必须有后果

用户的选择不能只改变文案语气，必须真实影响隐藏状态、后续节点变体、对方攻防强度和结局倾向。

### 3.4 主干固定，分叉局部

第一版采用：

`14 个主干节点 + 4 个关键分叉点 + 分叉后回主干`

避免无限发散，保证实现成本可控。

### 3.5 自有工作流优先

核心推演、状态变化、节点跳转、复盘结算必须掌握在自有后端中。  
元器只读取结构化产物进行演示和映射。

## 4. 与现有代码基线的关系

当前代码中已经存在可复用的底座，但尚未形成真正的文游引擎。

现有可复用部分：

- `apps/api/app/orchestrators/trial_state_machine.py`
  现有顺序状态机，可保留为阶段规则基础层
- `apps/api/app/routes/simulation.py`
  可保留为庭审推演路由入口
- `apps/api/app/routes/analysis.py`
  可保留为分析与报告路由入口
- `apps/api/app/services/yuanqi_bridge.py`
  继续作为元器演示层桥接器，但不再承担核心推演职责
- `apps/api/app/schemas/case.py`
  继续作为案件画像基础输入
- `apps/api/app/schemas/turn.py`
  将扩展为运行态快照结构的一部分
- `apps/api/app/schemas/analysis.py`
  可保留为复盘、对方画像、胜诉率输出基础模型

本设计不推翻现有项目，而是在现有基础上新增一层“规则驱动的庭审文游引擎”。

## 5. 总体架构

### 5.1 架构结论

推荐采用：

`固定主轴编排器 + 文游节点引擎 + 分析节点 + 元器演示适配层`

### 5.2 逻辑分层

第一层：案件基础层

- 负责案件录入、案件标准化、争点抽取、证据缺口整理

第二层：庭审文游引擎层

- 负责节点选择
- 负责选项执行
- 负责隐藏状态变化
- 负责节点变体切换

第三层：分析与报告层

- 负责对方画像
- 负责趋势分析
- 负责结局评估
- 负责复盘报告

第四层：展示与适配层

- Web 前端渲染为沉浸式庭审页面
- 元器读取结构化结果做演示流程

## 6. 核心运行对象

### 6.1 Case

静态案件档案。来源于案件录入模块。  
基础结构沿用 `CaseProfile`。

### 6.2 TrialRun

一次完整庭审推演实例。  
一个案件可以拥有多轮 TrialRun。

建议字段：

- `trial_run_id`
- `case_id`
- `run_index`
- `current_node_id`
- `current_stage`
- `selected_path`
- `state`
- `active_flags`
- `result_type`
- `report_id`

### 6.3 StageNode

当前正在运行的剧情节点。  
每个节点必须是结构化配置，而不是一段自由文本。

### 6.4 Choice

节点下的可选诉讼策略。  
Choice 负责触发状态变化并决定下一节点去向。

### 6.5 Event

事件池中的单个事件。  
包括突袭证据、对方抗辩、法官追问、调解框架等。

### 6.6 ReplayReport

一次 TrialRun 的结算报告。  
面向用户、评委和历史回放模块。

### 6.7 RunCheckpoint

关键回放点。  
支持从高价值分叉节点重新开局。

## 7. 14 节点主线

### 7.1 主线总图

```text
N01 开庭
 -> N02 权利义务与诉请确认
 -> N03 原告开场陈述
 -> N04 被告答辩
 -> N05 法官归纳争点
 -> N06 我方举证布局
 -> N07 对方突袭证据
 -> N08 程序性申请/异议处理
 -> N09 我方主辩
 -> N10 对方反击
 -> N11 法官高压追问
 -> N12 我方最后陈述
 -> N13 调解意向探测
 -> N14 调解/宣判结果
```

### 7.2 节点表

| 节点 ID | 阶段 | 节点名称 | 主要作用 |
|---|---|---|---|
| N01 | 开庭准备 | 开庭 | 建立法庭仪式感与正式开庭氛围 |
| N02 | 开庭准备 | 权利义务与诉请确认 | 锁定程序位置与诉请边界 |
| N03 | 法庭调查 | 原告开场陈述 | 建立我方故事线与第一印象 |
| N04 | 法庭调查 | 被告答辩 | 给出对方完整叙事与抗辩主线 |
| N05 | 法庭调查 | 法官归纳争点 | 将双方争执压缩为核心争点 |
| N06 | 举证质证 | 我方举证布局 | 决定先打哪条证据链 |
| N07 | 举证质证 | 对方突袭证据 | 第一高潮，制造不确定性 |
| N08 | 举证质证 | 程序性申请/异议处理 | 通过程序工具抢回节奏 |
| N09 | 法庭辩论 | 我方主辩 | 将事实、证据、法理整合成体系论证 |
| N10 | 法庭辩论 | 对方反击 | 第二高潮，集中攻击我方薄弱点 |
| N11 | 法庭辩论 | 法官高压追问 | 法官以关键问题检验体系是否自洽 |
| N12 | 最后陈述 | 我方最后陈述 | 最后一次收束法官印象 |
| N13 | 调解/宣判 | 调解意向探测 | 决定是否进入调解窗口 |
| N14 | 调解/宣判 | 调解/宣判结果 | 输出本轮结局与结算感 |

### 7.3 四个关键分叉点

第一版重点做强以下四个节点：

- `N07`
- `N08`
- `N10`
- `N11`

原因：

- `N07` 负责对方突袭与戏剧张力
- `N08` 负责把法律程序工具真正纳入玩法
- `N10` 负责让前面选择产生“收账”效果
- `N11` 负责构造真实法庭中的高压检验

## 9. 隐藏状态系统

### 9.1 状态列表

第一版固定 7 个隐藏状态：

- `evidence_strength`
- `procedure_control`
- `judge_trust`
- `opponent_pressure`
- `contradiction_risk`
- `surprise_exposure`
- `settlement_tendency`

所有状态区间均为 `0-100`。

### 9.2 通用初始值

- `evidence_strength = 50`
- `procedure_control = 50`
- `judge_trust = 50`
- `opponent_pressure = 50`
- `contradiction_risk = 35`
- `surprise_exposure = 20`
- `settlement_tendency = 40`

### 9.3 案件类型修正

劳动争议：

- `evidence_strength -8`
- `opponent_pressure +4`
- `contradiction_risk +8`
- `surprise_exposure +8`
- `settlement_tendency +4`

民间借贷：

- `evidence_strength +6`
- `contradiction_risk -4`
- `surprise_exposure +4`
- `settlement_tendency -6`

离婚纠纷：

- `evidence_strength -6`
- `opponent_pressure +6`
- `contradiction_risk +6`
- `settlement_tendency +18`

侵权责任：

- `evidence_strength -4`
- `opponent_pressure +2`
- `surprise_exposure +4`
- `settlement_tendency -2`

### 9.4 状态对前端的展示原则

前端默认不显示原始数值，只显示描述词。

例如：

- `judge_trust` 高：明显信服
- `surprise_exposure` 高：突袭造成重创

## 12. 文游文本渲染规则

### 12.1 统一输出格式

每个节点统一输出四段：

- `cg_caption`
- `court_progress`
- `pressure_shift`
- `choice_prompt`

标准格式如下：

```text
【CG画面：……】

【法庭进程：……】

【局势变化：……】

【你准备如何应对？】
A. …
B. …
C. …
D. …
```

### 12.2 写作规则

`CG画面`

- 负责画面感
- 长度建议 45-90 字
- 必须包含场景元素、角色动作、气氛词

`法庭进程`

- 负责程序说明
- 长度建议 35-75 字
- 必须明确当前动作的法律或程序意义

`局势变化`

- 负责风险提示
- 长度建议 20-50 字
- 只提示一个最关键得失点

`选择提示`

- 负责行动召回
- 保持简短清晰

### 12.3 风格约束

必须：

- 庄重、克制、有画面感
- 中短句为主
- 风险描述具体

禁止：

- 热血动漫风
- 夸张文学比喻
- 空泛 AI 腔
- 过度营销式修辞

## 13. 企鹅角色说话风格

### 13.1 法官企鹅

角色定位：

- 程序与秩序中心
- 语气稳、指令清晰、少情绪

### 13.2 书记员企鹅

角色定位：

- 程序记录者
- 只做核对、宣读、记录

### 13.3 原告代理企鹅

角色定位：

- 用户策略代言人
- 条理清楚，专业但不僵硬

### 13.4 被告律师企鹅

角色定位：

- 压力制造者
- 冷静、精明、专抓漏洞

### 13.5 原告当事人企鹅

角色定位：

- 事实与情感补充来源
- 直白、真切，但不过度情绪化

### 13.6 证人企鹅

角色定位：

- 不确定性来源
- 语言应具体、朴素、有记忆边界感

### 13.7 调解语境下的法官企鹅

角色定位：

- 保持权威，但语气略缓
- 提供调解框架，不强迫达成和解

## 14. 结局判定

### 14.1 中间指标

第一版使用两个内部指标：

- `advantage_core`
- `settlement_window`

### 14.2 advantage_core

主要由以下状态组成：

- 正向：`evidence_strength`、`judge_trust`、`procedure_control`
- 负向：`contradiction_risk`、`surprise_exposure`、`opponent_pressure`

建议实现思路：

`0.35*evidence_strength + 0.30*judge_trust + 0.20*procedure_control - 0.10*contradiction_risk - 0.05*surprise_exposure`

### 14.3 settlement_window

主要由以下状态组成：

- 正向：`settlement_tendency`、`judge_trust`
- 负向：`opponent_pressure`、`contradiction_risk`

建议实现思路：

`0.45*settlement_tendency + 0.20*judge_trust - 0.20*opponent_pressure - 0.15*contradiction_risk`

### 14.4 第一版结果类型

只保留三类：

- 调解成功
- 调解失败，择期宣判
- 倾向性结果反馈

### 14.5 倾向性结果反馈分档

- `advantage_core >= 62`：我方较优
- `46 <= advantage_core < 62`：胶着
- `advantage_core < 46`：风险偏高

修正规则：

- 若 `contradiction_risk >= 65`，结果下调一档
- 若 `judge_trust >= 75` 且 `evidence_strength >= 65`，结果可上调半档

## 15. 复盘报告设计

### 15.1 复盘目标

复盘报告必须同时服务：

- 用户回看
- 下一轮策略修正
- 历史案件沉淀
- 比赛展示

### 15.2 固定八个区块

1. 报告头部
2. 本轮主轴概览
3. 关键转折点
4. 庭审过程回顾
5. 证据与风险面板
6. 对方战术画像
7. 下一轮建议
8. 结果结算

### 15.3 存储原则

复盘报告必须结构化存储，并可附带 `report_markdown` 用于导出和展示。

## 16. 历史案件与回放设计

### 16.1 三层结构

- `Case`
- `TrialRun`
- `RunCheckpoint`

### 16.2 历史案件页目标

首页只保留两个入口：

- 新建案件
- 历史案件

历史案件页内部再展开：

- 案件列表层
- 案件详情层

### 16.3 关键回放点

第一版仅保存高价值回放点：

- `CP-N07`
- `CP-N08`
- `CP-N10`
- `CP-N11`

支持从关键节点重新推演，不要求从每个节点都可重开。

## 17. Web 前端呈现要求

前端不做聊天窗口，固定采用“庭审舞台页”：

- 顶部：六大阶段进度条
- 中部：CG式文本主舞台
- 右侧：争点、证据、风险、对方状态
- 底部：本节点 2-4 个策略选择按钮

每次节点推进时，页面只刷新：

- 当前阶段
- 主舞台文本
- 右侧上下文
- 底部选项

## 18. 元器边界

元器在本设计中的职责是：

- 展示结构化工作流结果
- 读取同一份节点与分析产物做演示
- 帮助完成比赛展示链路

元器不负责：

- 核心节点跳转判定
- 隐藏状态维护
- 结局规则计算
- 复盘主逻辑生成

## 19. 第一版验收标准

以下条件同时满足，视为第一版核心工作流达标：

1. 可基于一个标准案件创建完整 TrialRun
2. 可从 `N01` 一直推进到 `N14`
3. 六大法定阶段顺序严格正确
4. `N07`、`N10`、`N11` 至少存在可感知变体
5. 用户选择会显著影响隐藏状态
6. `N14` 可输出三类结局之一
7. 每轮可生成结构化复盘报告
8. 历史案件页可展示多轮推演记录
9. 至少支持关键节点重开
10. 元器可读取结果进行演示，但主逻辑不依赖元器

## 20. 实施优先级

第一优先级：

- 14 节点主干跑通
- 7 个隐藏状态生效
- 4 个关键节点变体可用
- 复盘报告生成

第二优先级：

- 事件池丰富
- 企鹅角色文本风格打磨
- 历史案件对比体验优化

第三优先级：

- 更多案件类型深扩
- 更细颗粒度分支
- 真 CG、立绘、额外视觉资源

## 21. 风险与边界提醒

### 21.1 最大风险

如果后续实现滑回“自由聊天式法律问答”，则本设计失效。  
必须坚持规则驱动的节点引擎架构。

### 21.2 第二风险

如果前端把全部信息铺满一页，会破坏文游沉浸感。  
必须坚持主舞台优先、侧栏辅助、底部选择的结构。

### 21.3 第三风险

如果元器反向控制主逻辑，会导致系统后续不可维护。  
必须坚持自有后端为主、元器为演示层。

## 22. 结语

本规格确定了企鹅法庭第一版最重要的产品形态：

`它不是一个通用法律聊天机器人，而是一套严格受法律程序约束的沉浸式庭审文游引擎。`

后续无论是 Web 前端、后端工作流、复盘模块，还是元器演示接入，都应以本设计为主轴，不得偏离。
