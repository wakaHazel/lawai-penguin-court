# 元器工作流详细设计（庭审模拟链路）

> 版本：2026-04-07  
> 适用范围：企鹅法庭·案件录入 -> 庭审模拟 -> 对方行为模拟 -> 胜诉率分析 -> 复盘报告  
> 目标：用于后续在腾讯元器中落地可解释、可控、可演示的工作流

## 1. 先讲结论

这个项目里的元器，不应该被当成“随便接一个大模型聊天”的地方，而应该被当成：

`受控庭审工作流编排器`

也就是：

- 后端负责：案件结构、状态机、持久化、规则兜底
- 元器负责：场景生成、节点编排、工具调用、变量聚合、结构化回复

主原则是：

- `流程控制在后端`
- `内容生成在元器`
- `法律检索通过工具节点接得理 API`
- `最终返回结构化结果，不返回散乱长文本`

## 2. 为什么元器适合放在这里

这个项目最像元器的地方，不是“让它会说话”，而是：

- 每个庭审阶段都要有不同处理逻辑
- 不同案由要走不同 prompt 分支
- 某些阶段要先查法条，再查类案，再生成内容
- 多个节点的结果最后要聚合
- 输出要稳定，不能全靠大模型自由发挥

这正好适合元器的：

- 工具节点
- 大模型节点
- 代码节点
- 变量节点
- 聚合节点
- 回复节点

## 3. 整体主链路怎么拆

我建议在元器里，不是做一个超级大工作流，而是拆成 `4 条主工作流 + 1 条辅助工作流`：

### 3.1 工作流 A：庭审场景生成工作流

作用：

- 根据当前案件和当前阶段，生成这一轮应该展示的法庭场景
- 产出 `scene_title / scene_text / branch_options / next_stage_hint`

输入：

- `case_profile`
- `current_stage`
- `turn_index`
- `previous_action`
- `focus_issues`

输出：

- `scene_payload`

### 3.2 工作流 B：法律依据检索工作流

作用：

- 根据当前争议焦点去查法条和类案
- 给场景生成、胜诉率分析、复盘报告提供法律依据

输入：

- `case_type`
- `focus_issues`
- `fact_keywords`

输出：

- `law_search_result`
- `case_search_result`
- `legal_support_summary`

### 3.3 工作流 C：对方行为模拟工作流

作用：

- 模拟被告、对方律师、证人等在当前阶段的反应

输入：

- `opponent_profile`
- `current_stage`
- `case_type`
- `current_scene_payload`
- `available_opponent_strategies`

输出：

- `opponent_action`
- `opponent_argument`
- `surprise_evidence`
- `risk_delta`

### 3.4 工作流 D：胜诉率分析与报告工作流

作用：

- 汇总全流程数据，给出可解释分析和复盘结果

输入：

- `case_profile`
- `simulation_timeline`
- `legal_support_summary`
- `opponent_action_timeline`

输出：

- `claim_support_rate`
- `evidence_acceptance_rate`
- `focus_issue_win_rate`
- `risk_points`
- `advice_items`
- `report_markdown`

### 3.5 工作流 E：输入清洗与变量标准化辅助工作流

作用：

- 解决元器里面常见的变量结构不稳问题
- 尤其是数组、空值、字段缺失、中文标签与英文枚举转换

输入：

- 原始案件输入

输出：

- 标准化后的 `case_profile`

## 4. 元器里每一条工作流具体怎么搭

下面重点写第一版最该做的两条：

- `庭审场景生成工作流`
- `法律依据检索工作流`

这两条搭好了，项目主链就能演示。

---

## 5. 工作流 A：庭审场景生成工作流

### 5.1 目标

输入一个已经结构化的案件对象和当前阶段，输出这一轮庭审画面需要展示的内容。

### 5.2 推荐节点顺序

#### 节点 1：开始节点

接收后端传来的变量：

- `case_id`
- `case_profile`
- `current_stage`
- `turn_index`
- `selected_action`

#### 节点 2：变量标准化节点

作用：

- 把后端传来的字段重新整理成元器内部固定变量名
- 把可能不存在的字段补默认值

例如输出：

- `v_case_title`
- `v_case_type`
- `v_current_stage`
- `v_focus_issues`
- `v_claims`
- `v_evidence_summary`

#### 节点 3：代码节点

作用：

- 处理数组和空值
- 避免元器后续节点拿到 `null`

这里建议做：

- 把 `focus_issues` 转成字符串数组
- 把 `claims` 转成字符串数组
- 把空数组统一输出为 `[]`
- 把空字段统一替换成空字符串

原因：

- 得理 API 和后续大模型节点更喜欢稳定参数
- 元器变量聚合时最怕 `null`

#### 节点 4：阶段分流节点

根据 `current_stage` 分流：

- `prepare`
- `investigation`
- `evidence`
- `debate`
- `final_statement`
- `mediation_or_judgment`
- `report_ready`

每个阶段后面都接不同 prompt 节点。

#### 节点 5：大模型节点（按阶段分别设置）

每个阶段一套 prompt 模板。

例如：

- `prepare_scene_prompt`
- `investigation_scene_prompt`
- `evidence_scene_prompt`
- `debate_scene_prompt`

输出必须限定为 JSON 结构。

例如统一要求：

```json
{
  "scene_title": "",
  "scene_text": "",
  "speaker_roles": [],
  "branch_options": [],
  "next_stage_hint": ""
}
```

#### 节点 6：回复格式校验代码节点

作用：

- 检查返回值里有没有缺字段
- 如果模型少返回某个字段，就补默认值

例如：

- 没有 `branch_options` 就补空数组
- 没有 `scene_title` 就按阶段默认值补

#### 节点 7：回复节点

统一输出 `scene_payload`

### 5.3 这条工作流里的 prompt 应该怎么约束

核心要求：

- 不允许改变案件事实
- 不允许跳阶段
- 不允许发明不存在的关键证据
- 必须保持法庭语气
- 必须给出 2 到 4 个可选分支

#### 示例 prompt 框架

```text
你是“企鹅法庭”的庭审场景生成器。

你的任务不是自由聊天，而是根据当前案件信息与庭审阶段，
生成一轮受控的庭审场景内容。

你必须遵守以下规则：
1. 不得改变已给定案件事实
2. 不得跳过当前阶段
3. 不得凭空制造关键证据
4. 输出必须是 JSON
5. branch_options 只能给 2-4 个

当前案件类型：{{v_case_type}}
当前阶段：{{v_current_stage}}
案件摘要：{{v_case_summary}}
争议焦点：{{v_focus_issues}}
用户上一轮动作：{{v_selected_action}}

请输出：
{
  "scene_title": "",
  "scene_text": "",
  "speaker_roles": [],
  "branch_options": [],
  "next_stage_hint": ""
}
```

## 6. 工作流 B：法律依据检索工作流

### 6.1 目标

根据当前争议焦点，先查法条，再查类案，再生成一个简短法律依据摘要。

### 6.2 推荐节点顺序

#### 节点 1：输入节点

输入：

- `case_type`
- `focus_issues`
- `fact_keywords`

#### 节点 2：代码节点

作用：

- 把 `focus_issues` 转成检索关键词
- 把数组变成得理插件能接受的参数结构

这里必须处理你前面说过的那个问题：

- `app_secret`
- API 参数类型
- 避免直接把数组以不兼容格式传过去

#### 节点 3：得理法条检索工具节点

接调用得理开放平台法条检索接口的插件。

输出：

- `law_search_raw`

#### 节点 4：得理类案检索工具节点

输出：

- `case_search_raw`

#### 节点 5：大模型摘要节点

作用：

- 把法条检索结果和类案检索结果压缩成前端和后续分析更容易使用的摘要

输出：

```json
{
  "relevant_laws": [],
  "relevant_cases": [],
  "legal_support_summary": ""
}
```

#### 节点 6：聚合节点

把：

- 原始法条结果
- 原始类案结果
- 摘要结果

聚成统一变量。

---

## 7. 工作流 C：对方行为模拟工作流

这条工作流放在第二阶段做，不是当前最先接。

推荐输入：

- `case_type`
- `current_stage`
- `opponent_profile`
- `current_scene_payload`
- `user_selected_action`

推荐输出：

```json
{
  "opponent_action": "",
  "opponent_argument": "",
  "surprise_evidence": [],
  "risk_delta": []
}
```

它的关键不是“越聪明越好”，而是“越受控越好”。

必须强约束：

- 不能脱离案件背景
- 不能生成完全离谱的新事实
- 惊喜证据必须从预定义池里选

---

## 8. 工作流 D：胜诉率分析与复盘工作流

这条工作流放在主链稳定后接。

推荐节点顺序：

1. 输入案件与全流程时间线
2. 读法律依据摘要
3. 读证据表现摘要
4. 大模型评分节点
5. 报告结构化节点
6. 回复节点

输出必须不是单数字，而是：

```json
{
  "claim_support_rate": 0,
  "evidence_acceptance_rate": 0,
  "focus_issue_win_rate": 0,
  "risk_points": [],
  "advice_items": [],
  "report_markdown": ""
}
```

## 9. 元器和后端怎么配合

### 9.1 后端负责什么

后端负责：

- 保存 `case_profile`
- 保存当前阶段和回合数
- 决定能不能进入下一个阶段
- 决定调用哪条元器工作流
- 保存元器输出结果

也就是说，后端是“裁判员”，元器是“内容执行员”。

### 9.2 元器负责什么

元器负责：

- 生成当前场景
- 调用得理 API 工具节点
- 聚合检索结果
- 生成结构化场景或结构化分析结果

### 9.3 不能反过来

不能让元器负责：

- 自己决定整个庭审流程
- 自己决定是否跳阶段
- 自己保存长期业务状态
- 自己承担比赛主系统全部逻辑

否则后面会失控。

## 11. 得理 API 在元器里怎么接

推荐做法：

1. 先把得理 API 封装成元器插件
2. 至少提供两个工具：
   - `search_laws`
   - `search_cases`
3. 工具返回统一 JSON，不让大模型直接吃杂乱文本

### 工具返回建议格式

#### 法条检索

```json
{
  "items": [
    {
      "title": "",
      "article": "",
      "source": "",
      "summary": ""
    }
  ]
}
```

#### 类案检索

```json
{
  "items": [
    {
      "case_name": "",
      "court": "",
      "year": "",
      "summary": ""
    }
  ]
}
```

这样后面的模型节点会稳定很多。

## 12. 元器第一阶段最值得先搭哪几条

第一阶段只建议先搭：

1. `庭审场景生成工作流`
2. `法律依据检索工作流`

原因很简单：

- 它们最直接服务当前主链
- 也最容易做比赛演示
- 不需要先把整套系统全部搭完

对方行为模拟和胜诉率分析放第二阶段。

## 13. 我建议的实际落地顺序

### 第一步

后端把当前已有接口稳定下来：

- `POST /api/cases`
- `GET /api/cases/{case_id}`
- `POST /api/cases/{case_id}/simulate/start`
- `POST /api/cases/{case_id}/simulate/turn`

### 第二步

先在元器里搭 `庭审场景生成工作流` 的空骨架：

- 开始节点
- 变量标准化节点
- 阶段分流节点
- 每阶段一个大模型节点
- 回复节点

### 第三步

把得理插件接进 `法律依据检索工作流`

### 第四步

后端写一个 `yuanqi_bridge`

它只负责：

- 调哪条元器工作流
- 传哪些变量
- 收什么结构化返回

### 第五步

前端庭审页接元器返回的 `scene_payload`

## 14. 第一版演示时你该怎么讲

最稳的讲法不是：

“我们做了一个很厉害的多智能体系统。”

而是：

“我们基于腾讯元器把庭审过程拆成受控工作流节点，
把法律检索接入得理开放平台 API，
由后端状态机严格控制法定流程，
最终实现案件录入、庭审模拟、法律支撑、分析报告这一条完整链路。”

这个讲法既贴赛题，也稳。
