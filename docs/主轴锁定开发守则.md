# 主轴锁定开发守则

> 版本：2026-04-07  
> 适用项目：`企鹅法庭·沉浸式庭审模拟与立法沙盘推演AI智能体系统`

## 1. 目的

这份守则的唯一目的，是防止项目在开发过程中跑偏、扩题、弱化赛题主线，或者被技术实现细节带离核心价值。

以后所有开发动作，默认都要先服从这份守则，再讨论实现方式。

## 2. 项目主轴

本项目的唯一主轴固定为：

`案件录入 -> 庭审模拟 -> 对方行为模拟 -> 胜诉率分析 -> 复盘报告`

所有新增模块、页面、接口、工作流、Prompt、演示内容，都必须直接服务这条主轴。

如果某项工作不能明确映射到这条主轴，就默认：

- 不做
- 延后
- 不进入 MVP

## 3. 三层最高约束

后续开发必须同时服从以下三层约束，优先级从高到低如下：

### 第一层：赛题约束

以 [D06正式赛题要求摘要.md](/E:/lawai/docs/D06正式赛题要求摘要.md) 为准。

硬约束包括：

- 赛题必须是 `法律AI应用创新与实践`
- 项目定位必须是 `系统解决方案`
- 官方工具主叙事必须围绕这 6 项：
  - 腾讯开悟比赛平台
  - CodeBuddy
  - 腾讯元器
  - 腾讯元宝
  - 小理AI
  - 得理开放平台API

### 第二层：产品约束

以 [企鹅法庭项目技术方案V1.md](/E:/lawai/docs/企鹅法庭项目技术方案V1.md) 为准。

硬约束包括：

- 核心场景必须是 `沉浸式庭审模拟`
- 三个核心智能体必须保持：
  - 庭审流程交互智能体
  - 对方行为模拟智能体
  - 胜诉率分析智能体
- MVP 优先案由保持：
  - 民间借贷
  - 劳动争议

### 第三层：实施约束

以 [2026-04-06-企鹅法庭MVP实施计划.md](/E:/lawai/docs/superpowers/plans/2026-04-06-企鹅法庭MVP实施计划.md) 为准。

硬约束包括：

- 先做主链闭环，再补细节
- 先做可演示 MVP，再做增强
- 不并行扩展无关模块

## 4. 开发顺序铁律

以后开发顺序固定如下，不得随意跳跃：

1. `案件录入骨架`
2. `案件结构化数据模型`
3. `庭审状态机骨架`
4. `元器工作流骨架`
5. `得理开放平台API接入口`
6. `对方行为模拟`
7. `胜诉率分析`
8. `复盘报告输出`
9. `演示优化`

解释：

- 没有案件录入，就没有后续所有动作
- 没有状态机，庭审模拟就会失控
- 没有法律检索支撑，分析模块会失真
- 没有报告输出，主链就不完整

## 5. 每次开工前必须回答的 4 个问题

以后每次开始一个新任务，必须先明确以下四项：

### 5.1 今天只做什么

必须是一个具体模块，不能写空话。

例如：

- 今天只做案件录入页
- 今天只做案件 schema
- 今天只做元器工作流占位

### 5.2 它服务主轴的哪一段

必须映射到主轴中的一个环节。

例如：

- 服务 `案件录入`
- 服务 `庭审模拟`
- 服务 `胜诉率分析`

### 5.3 今天明确不做什么

必须写出边界，防止越做越散。

例如：

- 不做登录
- 不做后台管理
- 不做立法沙盘
- 不做多角色权限

### 5.4 什么叫完成

必须有可验证的完成标准。

例如：

- 能成功录入 4 类 MVP 案由
- 能从结构化案件数据启动一次模拟
- 能输出一版胜诉率分析 JSON

## 6. 一次只允许做一种类型的任务

为了防止散掉，开发时一次只做以下一种任务：

- `页面骨架`
- `数据结构`
- `接口骨架`
- `工作流骨架`
- `规则表`
- `输出模板`

禁止在一个回合里混做太多东西，例如：

- 一边做页面，一边扩案由，一边改答辩材料
- 一边改工作流，一边做后台权限，一边加立法沙盘

## 7. MVP 明确允许做的事情

以下内容属于允许进入当前 MVP 的范围：

- 案件信息录入
- 庭审阶段推进
- 对方抗辩与行为模拟
- 法规与类案检索
- 胜诉率分析
- 复盘报告生成
- 庭前备战清单
- 4 个高频案由的演示样本

## 8. MVP 明确不允许提前做的事情

以下内容默认不进入当前阶段，除非你明确点名让我做：

- 泛法律聊天机器人
- 通用多 agent 平台
- 复杂用户系统
- 运营后台
- 多租户
- 社区功能
- 大而全案由扩张
- 深度立法沙盘
- 炫技型复杂架构

## 9. 技术表达与比赛表达必须分开

开发时可以有常规工程实现，但对外主叙事不能被工程细节带偏。

必须始终区分：

- `比赛表达`
  - 腾讯开悟比赛平台
  - CodeBuddy
  - 腾讯元器
  - 腾讯元宝
  - 小理AI
  - 得理开放平台API
- `工程实现`
  - 前端框架
  - 后端框架
  - 本地数据存储
  - 模板与缓存

原则是：

- 对外先讲比赛表达
- 对内再讲工程实现

## 10. 三条不过线规则

后续开发默认永远遵守这三条：

1. 任何新模块，如果不能直接服务主链，就不做。
2. 任何新设计，如果不能映射到 D06 交付项，就不写。
3. 任何技术选择，如果会削弱“腾讯系AI工具平台”主叙事，就不用它做项目主表达。

## 11. 任务下发模板

以后你给我布置任务，建议直接用这个格式：

```text
今天只做：
服务主轴的哪一段：
明确不做：
完成标准：
```

示例：

```text
今天只做：案件录入页和案件结构化字段
服务主轴的哪一段：案件录入
明确不做：登录、后台管理、立法沙盘、答辩材料
完成标准：能录入 4 类 MVP 案由并保存成统一结构
```

## 12. 守则执行方式

以后如果你说：

- `按守则继续`
- `按主轴继续`
- `不要跑偏，按守则做`

我都默认按这份文档执行，不再自由扩题。

## 13. 最后的判断标准

以后判断一项工作该不该做，只看一句话：

`它能不能直接强化“案件录入 -> 庭审模拟 -> 对方行为模拟 -> 胜诉率分析 -> 复盘报告”这条主链？`

如果答案是：

- `能`：继续做
- `不能`：先不做
