# 元器工作流联动契约（企鹅法庭）

> 版本：2026-04-07  
> 适用项目：企鹅法庭·沉浸式庭审模拟与立法沙盘推演 AI 智能体系统  
> 目的：把“谁产出什么、谁消费什么、谁有最终决定权”写死，避免元器接入时返工

## 1. 这份文档是干什么的

这份文档不是操作手册，而是工作流联动契约。

它只回答 5 件事：

1. 正式联调时，到底是一回合调一次主控，还是每个子工作流独立调。
2. 后端、元器、得理插件，谁对哪些字段拥有最终决定权。
3. `W01 / W02 / W03 / W04` 之间怎么一环扣一环。
4. 哪些字段可以回写系统状态，哪些字段只能展示。
5. 哪些失败可以降级继续，哪些失败必须阻断。

如果这份文档和 [元器接入实操手册（企鹅法庭）.md](/E:/lawai/docs/元器接入实操手册（企鹅法庭）.md) 有冲突，以这份联动契约为准。

## 2. 正式模式和调试模式必须分开

### 2.1 正式联调模式

正式联调和比赛演示时，必须采用：

`一回合一次 W00 主控编排`

也就是说，后端每推进一个庭审回合，只发起一次 `W00_企鹅法庭主控编排` 调用。

然后由 `W00` 按阶段顺序继续调内部子工作流：

- `W01_庭审场景生成`
- `W02_法律支持检索`
- `W03_对方行为模拟`
- `W04_结果分析复盘`

### 2.2 调试模式

只有在单独排查节点问题时，才允许：

- 只调 `W01`
- 只调 `W02`
- 只调 `W03`
- 只调 `W04`

调试模式的结果不能直接视为正式链路设计。

## 3. 工作流模式选择

根据当前项目主轴，推荐模式如下：

| 工作流 | 模式 | 原因 |
| --- | --- | --- |
| `W00_企鹅法庭主控编排` | `orchestrator + router` | 需要按阶段和任务类型统一调度子工作流 |
| `W01_庭审场景生成` | `sequential` | 输入固定，输出稳定，强调结构化场景生成 |
| `W02_法律支持检索` | `sequential` | 先整理关键词，再调插件，再整理结果 |
| `W03_对方行为模拟` | `sequential` | 使用结构化案件和法律上下文生成对方行为 |
| `W04_结果分析复盘` | `sequential + validation` | 需要消费前置结果，并做输出完整性校验 |

## 4. 字段所有权必须写死

### 4.1 后端拥有最终决定权的字段

这些字段在当前代码和第一阶段接入里，都只能由后端决定：

- `current_stage`
- `turn_index`
- `selected_action`
- `available_actions`
- `simulation_id`
- `case_id`
- `workflow_key`

解释：

- 当前后端已经在 [simulation.py](/E:/lawai/apps/api/app/routes/simulation.py) 中校验 `selected_action` 是否属于当前快照的 `available_actions`
- 所以元器第一阶段不能成为 `available_actions` 的最终来源

### 4.2 元器拥有最终决定权的字段

这些字段可以由元器产出，并作为展示或分析结果：

- `scene_title`
- `scene_text`
- `legal_support_summary`
- `recommended_laws`
- `recommended_cases`
- `opponent_action`
- `opponent_argument`
- `risk_delta`
- `win_rate_estimate`
- `issue_assessment`
- `risk_points`
- `evidence_gaps`
- `next_steps`
- `report_markdown`

### 4.3 共享但受枚举约束的字段

这些字段可以由元器返回，但必须严格输出后端规范值：

- `speaker_role`
- `opponent_role`

当前允许值以 [common.py](/E:/lawai/apps/api/app/schemas/common.py) 为准：

- `plaintiff`
- `defendant`
- `applicant`
- `respondent`
- `agent`
- `witness`
- `judge`
- `other`

如果元器输出自然语言标签，例如“审判长”“被告代理律师”，都视为不合格输出。

## 5. 第一阶段必须接受的现实约束

为了避免返工，第一阶段先锁定下面这条边界：

### 5.1 `W01` 不负责最终动作判定

`W01_庭审场景生成` 可以输出：

- `speaker_role`
- `suggested_actions`
- `next_stage_hint`

但第一阶段不能直接回写：

- `available_actions`

原因：

- 当前系统的 `available_actions` 仍由后端状态机生成
- 如果现在让元器也生产 `available_actions`，会形成双源冲突

所以第一阶段应该这样理解：

- 后端 `available_actions`：正式可点击动作
- 元器 `suggested_actions`：展示型建议动作，仅用于文案增强和后续设计参考

## 6. 正式一回合闭环应该怎么跑

### 6.1 `prepare / investigation / evidence`

正式顺序：

`后端 -> W00 -> W01 -> W02 -> W00 合并 -> 前端展示`

本阶段不调用：

- `W03`
- `W04`

### 6.2 `debate / final_statement`

正式顺序：

`后端 -> W00 -> W01 -> W02 -> W03 -> W00 合并 -> 前端展示`

本阶段：

- `W02` 必须先于 `W03`
- 因为对方行为模拟必须尽量参考当前争议点对应的法律支持，而不是空想

### 6.3 `mediation_or_judgment / report_ready`

正式顺序：

`后端 -> W00 -> W01 -> W02 -> 可选 W03 上下文 -> W04 -> W00 合并 -> 前端展示`

这里的关键规则是：

- `W04` 必须消费 `W02` 产出的法律支持摘要
- `W04` 可以消费 `W03` 的对方行为摘要，但它不是强制前置

## 7. 子工作流之间的消费关系

### 7.1 `W01` 的产物

`W01` 输出：

- `scene_title`
- `scene_text`
- `speaker_role`
- `suggested_actions`
- `branch_focus`
- `next_stage_hint`

它的产物用途：

- 给前端展示
- 给 `W03` 提供当前回合场景语境
- 给 `W04` 提供当前回合叙事焦点

### 7.2 `W02` 的产物

`W02` 输出：

- `legal_support_summary`
- `recommended_laws`
- `recommended_cases`
- `issue_mapping`
- `missing_points`

它的产物用途：

- 给前端展示
- 给 `W03` 提供对方抗辩时的法律上下文
- 给 `W04` 作为强制输入

### 7.3 `W03` 的产物

`W03` 输出：

- `opponent_action`
- `opponent_argument`
- `opponent_evidence`
- `risk_delta`
- `next_pressure`

它的产物用途：

- 给前端展示
- 给 `W04` 作为可选增强输入

### 7.4 `W04` 的产物

`W04` 输出：

- `win_rate_estimate`
- `issue_assessment`
- `risk_points`
- `evidence_gaps`
- `next_steps`
- `report_markdown`

它的产物用途：

- 给复盘页展示
- 给报告导出
- 给比赛材料中的样例输入输出使用

## 8. `W04` 的前置条件必须明确

`W04_结果分析复盘` 不能只吃案件基础字段，它的正式输入必须包括：

- `case_id`
- `case_type`
- `current_stage`
- `turn_index`
- `branch_focus`
- `focus_issues_json`
- `claims_json`
- `missing_evidence_json`
- `opponent_arguments_json`
- `legal_support_summary`
- `recommended_laws_json`
- `recommended_cases_json`
- `issue_mapping_json`
- `opponent_action`
- `opponent_argument`
- `opponent_evidence_json`
- `risk_delta`

其中：

- `legal_support_summary`、`recommended_laws_json`、`recommended_cases_json`、`issue_mapping_json` 是强制前置
- `opponent_action`、`opponent_argument`、`opponent_evidence_json`、`risk_delta` 是可选增强

## 9. 如果前置结果丢了，怎么兜底

为了避免正式演示时中断，定义如下兜底规则：

### 9.1 `W02` 失败

后果：

- `W04` 不得给出强结论

允许输出：

```json
{
  "status": "degraded",
  "analysis_blocked_reason": "missing_legal_support",
  "next_steps": ["请先完成法律支持检索再生成复盘报告"]
}
```

### 9.2 `W03` 失败

后果：

- `W04` 仍可继续

但必须标记：

- `opponent_context_status = "missing"`

### 9.3 `W01` 失败

后果：

- 当前回合展示链路直接阻断

原因：

- 场景文本是所有下游展示的入口

## 10. 后端必须补的 3 层适配

如果不补这 3 层，工作流不会真正环环相扣。

### 10.1 `yuanqi_payload_adapter.py`

职责：

- 把当前 `workflow_hints.variables` 转成元器真正要吃的 `custom_variables`
- 统一把数组转成 JSON 字符串
- 做字段重命名，例如：
  - `v_focus_issues -> focus_issues_json`
  - `v_claims -> claims_json`
  - `v_missing_evidence -> missing_evidence_json`
  - `v_opponent_arguments -> opponent_arguments_json`

### 10.2 `yuanqi_context_store.py`

职责：

- 以 `simulation_id + turn_index + current_stage` 为键
- 缓存 `W02` 和 `W03` 结果
- 给 `W04` 提供正式输入

### 10.3 `yuanqi_response_merger.py`

职责：

- 把 `W01 / W02 / W03 / W04` 结果合并成统一回包
- 明确哪些字段回前端
- 明确哪些字段只进报告
- 明确哪些字段只进缓存

## 11. 元器凭据配置不要写死语法

手册里出现的：

- `{{DELILEGAL_APP_ID}}`
- `{{DELILEGAL_APP_SECRET}}`

只能理解为“占位写法”，不能当成元器后台唯一固定语法。

真正执行时，必须按元器后台实际 UI 去配置：

- 插件密钥字段
- 安全变量
- 请求头注入方式

也就是说：

- 得理 API 的认证字段是确定的
- 元器后台怎么保存和注入这些字段，要以实际配置界面为准

## 12. 你明天搭的时候，最稳的执行顺序

1. 先建 `得理法律检索插件（企鹅法庭）`
2. 再建 `W01_庭审场景生成`
3. 再建 `W02_法律支持检索`
4. 再建 `W03_对方行为模拟`
5. 再建 `W04_结果分析复盘`
6. 最后建 `W00_企鹅法庭主控编排`
7. 先用调试模式分别测 4 条子工作流
8. 再切到正式模式，只测“一回合一次 W00”

## 13. 一句话结论

真正不会出错、而且环环相扣的方案不是：

`每个 workflow_hint 独立调一次 W00`

而是：

`每个庭审回合只调一次 W00，由 W00 顺序驱动 W01 -> W02 -> W03/W04，并由后端缓存与合并关键结果`

这才是正式联调和比赛演示应该遵守的闭环。
