# AI工具使用说明（企鹅法庭）V1

## 1. 官方工具使用清单

本项目严格按照赛题要求，使用以下 6 项官方工具完成开发任务：

1. `腾讯开悟比赛平台`
2. `CodeBuddy`
3. `腾讯元器`
4. `腾讯元宝`
5. `小理AI`
6. `得理开放平台API`

## 2. 工具职责分工

### 2.1 腾讯开悟比赛平台

用于赛题资料获取、工具链路打通、项目提交与展示管理。

### 2.2 CodeBuddy

用于前端与后端代码开发、工程协同与项目联调。

### 2.3 腾讯元器

作为本项目智能体工作流编排主载体，负责组织案件识别、法律检索、庭审推进、对方行为模拟和结果输出等节点。

### 2.4 腾讯元宝

用于 Prompt 优化、庭审场景文案润色、报告文本表达优化与答辩材料辅助生成。

### 2.5 小理AI

用于法律实务案例参考、法律应用形态对比参考与项目输出表达辅助优化。

### 2.6 得理开放平台API

用于法规检索、案例检索和法规详情获取，为系统生成提供法律原子能力支撑。

## 3. 元器工作流设计

### 3.1 工作流目标

围绕“案件录入 -> 庭审模拟 -> 对方行为模拟 -> 胜诉率分析 -> 报告生成”形成可控的法律 AI 工作流。

### 3.2 工作流主链路

1. 用户输入案件基础信息
2. 元器识别案件类型与核心争点
3. 工具节点调用得理开放平台API进行法规与类案检索
4. 大模型节点生成当前庭审阶段的互动文本
5. 对方行为模拟节点生成抗辩、质疑或突发情况
6. 聚合节点汇总案件状态、证据变化和风险提示
7. 输出节点返回当前阶段建议与胜诉率分析结果

### 3.3 工作流节点设计

| 节点 | 类型 | 作用 |
|---|---|---|
| 案件识别节点 | 大模型节点 | 判断案件类型与争点 |
| 法规检索节点 | 工具节点 | 调用得理 API 检索法规 |
| 类案检索节点 | 工具节点 | 调用得理 API 检索案例 |
| 庭审生成节点 | 大模型节点 | 生成当前阶段庭审内容 |
| 对方模拟节点 | 大模型节点 | 生成对方抗辩与行为 |
| 聚合节点 | 变量聚合/代码节点 | 汇总多路结果并避免空值 |
| 输出节点 | 回复节点 | 输出页面展示内容 |

## 4. Prompt 设计说明

### 4.1 Prompt 总体原则

- 围绕法律场景，不脱离案件事实
- 以结构化输出为主，便于前端渲染与后端校验
- 引导模型对证据不足与结论风险进行提示
- 明确系统定位为法律辅助，不输出绝对司法结论

### 4.2 核心 Prompt A：庭审流程交互

目标：

- 生成当前庭审阶段文本
- 输出用户可选策略分支
- 保持阶段推进可控

关键要求：

- 不跳过法定阶段
- 不虚构关键事实
- 输出字段固定

### 4.3 核心 Prompt B：对方行为模拟

目标：

- 模拟对方当事人或代理人的典型抗辩
- 提升案件互动的真实性

关键要求：

- 仅在案件设定与证据范围内生成
- 不制造脱离法律逻辑的冲突

### 4.4 核心 Prompt C：胜诉率分析

目标：

- 综合争点、证据、法规与类案结果输出分析

关键要求：

- 分维度解释
- 输出风险点与补证建议
- 不使用“保证胜诉”等绝对表述

## 5. 得理开放平台API接入说明

本项目通过得理开放平台API获取法律原子能力，重点使用：

- 类案检索接口
- 法规检索接口
- 法规详情接口

接入方式上，优先在腾讯元器工作流中通过工具节点调用；同时由后端对结果进行缓存、聚合与兜底处理。

## 6. 小理AI使用说明

小理AI在本项目中主要承担法律实务应用参考角色，帮助团队参考法律问答、AI 检索、合同生成与审查、文本分析、文书生成等法律 AI 应用形态，并辅助优化项目输出的法律表达方式。

## 7. 风险控制

为保证法律专业性与输出稳健性，本项目设置以下约束：

- 关键法律内容由法学同学人工审校
- 得理开放平台API检索结果作为法律支撑来源之一
- 小理AI仅作辅助参考
- 高风险输出采用结构化约束与模板兜底
- 所有分析结果均强调辅助性质，不替代正式法律意见
