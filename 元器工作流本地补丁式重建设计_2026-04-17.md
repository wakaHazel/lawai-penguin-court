# 元器工作流本地补丁式重建设计

**日期：** 2026-04-17  
**项目：** 企鹅法庭  
**范围：** 基于最新元器导出包，对 `W00 / W02 / W04` 做本地补丁式修复与重打包，生成新的可导入 zip  
**目标：** 不从零重建工作流，而是在最新真实导出结构上修补关键断点，得到一份更接近元器真实运行环境、可继续交给队友导入调试的本地版本。

## 1. 背景与问题定义

当前工作流并不是“完全不能导入”，而是存在一组会导致“节点执行成功但最终无可见输出”或“上游节点有结果但下游拿不到字段”的结构性问题。  
这些问题集中在最新导出包 `tmp/yuanqi_export_W02_2026-04-17` 中的三条关键链路：

- `W00` 主控编排链路
- `W02` 法律支持检索链路
- `W04` 胜诉率分析与复盘链路

已确认的典型症状包括：

- `W00` 某些分支不再通过 `ANSWER` 节点对用户回传内容，导致工作流内部有结果但聊天侧 `content = null`
- `W00` 分支汇总代码仍把多个分支结果直接拼接，而不是选择命中的单一路径结果
- `W00` 部分 LLM 节点缺失 `Outputs` 声明，导致引用 `Output.Content` 时存在结构性不稳定
- `W02` 的输入预处理代码对 JSON 数组字符串兼容不足，容易把本应是数组的参数当成单段字符串处理
- `W02` 检索插件分页偏大，调试时冗余信息过多，不利于稳定回传
- `W04` 的 LLM 节点缺失显式输出 schema，导致结束节点虽然引用 `Output.Content`，但上游声明并不完整

本次设计不解决“元器线上环境中的所有发布配置问题”，而是解决“本地导出包结构不稳、字段不齐、链路收口不清”的问题。

## 2. 方案选择

本次有三种可行路线：

### 方案 A：基于最新真实导出包做补丁式重建

直接以 `tmp/yuanqi_export_W02_2026-04-17` 为底稿，逐个修改目标工作流 JSON，再重新打包成新的导入 zip。

优点：

- 最大限度保留你朋友当前在元器里已经调过的真实结构
- 与元器导出格式最接近，最不容易产生“本地拼出来能看但元器不认”的偏差
- 修改范围清晰，只修关键断点，返工成本最低

缺点：

- 需要对现有导出 JSON 做定点修复，补丁逻辑要写得谨慎

### 方案 B：改造旧生成脚本，重新吐一整套包

优点是可重复生成，缺点是旧脚本基于较早期的结构，和你朋友当前导出的真实包已有漂移，容易再次出现“脚本认为正确、元器实际不认”的情况。

### 方案 C：从 sample 重新拼装

优点是结构最干净，缺点是最脱离真实导出格式，风险最高。

**结论：选择方案 A。**

## 3. 重建边界

本次本地补丁式重建只覆盖以下内容：

### 3.1 必修内容

- 修复 `W00` 的主控分支收口问题
- 修复 `W00` 中缺失或不稳定的输出声明
- 修复 `W02` 的输入标准化逻辑
- 修复 `W04` 的输出 schema 缺失问题
- 重新生成新的导入 zip
- 对新包做静态审计，确认字段引用闭环

### 3.2 本次不处理的内容

- 不直接替你修改元器线上发布页配置
- 不替代你朋友在元器后台做最终联调与发布
- 不在本次强行重写所有插件密钥配置方式
- 不试图修复所有历史版本 zip，只产出一个新的本地可导入版

## 4. 目标产物

本次设计的最终交付物有两类：

### 4.1 导入产物

- 一个新的本地元器导入包 zip
- 一个对应的解包目录，便于继续人工核查

### 4.2 审计产物

- 一份针对新 zip 的静态审计结果
- 一份修复说明，列出改了哪些节点、为什么改、预期解决什么问题

## 5. 工作流级设计

### 5.1 W00 主控编排

`W00` 是整个系统的总入口，决定不同阶段请求如何流向 `W01 / W02 / W03 / W04`。

本次要把它修成“单分支命中、单结果回传、结构闭环”的形态，具体目标如下：

- 保持现有逻辑判断分支不大改，避免破坏你朋友当前在线上能识别的阶段条件
- 修复分支末端的结果回传收口，保证每条合法路径最终都能形成可被结束节点消费的统一输出
- 汇总节点不再把多个分支的 `final_out` 和 `branch_name` 直接字符串拼接，而是按优先顺序选择第一个非空结果
- 对缺失 `Outputs` 的 LLM 节点补齐 `Output.Content` 与 `Output.Thought` 声明，确保下游引用稳定
- 保留 fallback 分支，但 fallback 只作为兜底，不再混入正常分支聚合逻辑

这意味着 `W00` 的职责是：

- 根据 `current_stage` 选路
- 从命中的子工作流中接收标准结果
- 统一转换成结束节点可消费的 `final_out / branch_name`

### 5.2 W02 法律支持检索

`W02` 的核心问题不是插件完全不可用，而是输入标准化过弱，容易把本应为数组的字段当成普通字符串。

本次修复目标如下：

- 让输入预处理代码优先识别 JSON 数组字符串
- 如果 JSON 解析失败，再退回到逗号切分
- 对空值给出稳定默认值，避免后续插件拿到空数组时报错或失真
- 将检索分页从较大的调试量收敛到更适合回传稳定性的范围

`W02` 修复后应保证：

- `focus_issues_json` 和 `fact_keywords_json` 无论来自元器变量面板还是 API 透传，都能被尽量稳定转成数组
- `search_case` 与 `search_law` 获得的参数类型与预期一致
- LLM 汇总节点继续输出 `Output.Content`

### 5.3 W04 胜诉率分析与复盘

`W04` 当前最明显的问题是 LLM 节点输出 schema 不完整，结束节点引用了 `Output.Content`，但上游未显式声明。

本次修复目标如下：

- 为核心 LLM 节点补齐标准输出 schema
- 保持现有 prompt 和输入字段不大改，避免改变你们已经调过的业务表达
- 保证结束节点 `report_payload` 引用路径稳定成立

## 6. 数据流设计

本次补丁后的数据流原则只有一条：**每个被引用字段，都必须在上游节点显式存在。**

具体约束如下：

- `START` 节点输入字段不强行重命名，尽量沿用当前真实导出包字段名
- `LLM` 节点凡是被下游引用 `Output.Content` 的，必须存在 `Outputs -> Output -> Content`
- `CODE_EXECUTOR` 节点输出字段必须与结束节点或后续节点引用路径一一对应
- `END` 节点只引用已经在直接上游显式声明的字段

这次不追求“最漂亮的数据模型”，只追求“引用闭环明确、字段不悬空”。

## 7. 错误处理设计

本次错误处理不做复杂扩展，只做最实用的兜底：

- `W00` fallback 分支保留，保证阶段不匹配时仍有可回传文本
- 分支聚合节点在所有主分支都为空时才回退到兜底分支
- `W02` 输入标准化代码在解析失败时不抛异常，而是进入降级切分逻辑
- 静态审计阶段把“引用不存在字段”“LLM 缺失输出 schema”“代码节点语法错误”作为阻断项

## 8. 验证设计

本次验证分为两层：

### 8.1 结构验证

基于脚本审计以下问题：

- 是否存在被引用但未声明的 `Output.*`
- 是否存在缺失输出声明的关键 LLM 节点
- 代码节点是否能通过 Python 编译
- 插件依赖是否仍然存在硬编码头部字段

### 8.2 交付验证

检查新生成的 zip 是否满足：

- 可正常解压
- 关键工作流 JSON 已被替换
- 文件组织仍保持元器导出包风格

本次不在本地模拟元器云端真实执行，只做“导入前结构稳定性验证”。

## 9. 文件与脚本策略

本次不会从零新造一整套元器包结构，而是采用以下策略：

- 复制最新导出目录到新的工作目录
- 新增一个专用补丁脚本，对 `W00 / W02 / W04` 做定点修改
- 新增或复用审计脚本，对补丁结果做静态检查
- 最后再打包成新的 zip

这样做的原因是：

- 可追溯
- 可重复
- 可继续迭代

## 10. 风险与对应控制

### 风险 1：导出包 JSON 结构与旧脚本假设不一致

控制方式：不用旧脚本直接全量覆盖，而是做“读取真实包后定点修改”。

### 风险 2：修完结构后，元器线上仍因发布配置问题跑不通

控制方式：本次交付明确标记为“本地结构修复版”，不把线上发布配置问题混进本次修复结论。

### 风险 3：修复时误改到你朋友已在线上调通的业务逻辑

控制方式：只修 schema、分支收口、字段标准化，不大改 prompt 和主业务节点顺序。
